# brain/

The "what Hamzaish knows" layer.

## Two kinds of memory live here

### 1. Markdown — the source of truth

Everything is plain markdown files, organized by kind:

- **`identity/`** — who Hamza is, working preferences, role
- **`learnings/`** — append-only retro log (`YYYY-MM-DD.md`), what worked / what didn't / what surprised
- **`anti-patterns/`** — things we've decided not to do, with the reason
- **`decision-log/`** — cross-product strategic decisions
- **`knowledge/`** — ingested external knowledge (notes, papers, links)
- **`persona.md`** — voice and decision style
- **`operating-principles.md`** — the rules we don't violate
- **`weekly-review-template.md`** — the template for weekly self-review

You can read, edit, and commit these directly. They are the source of truth.

### 2. SQLite — the derived index

`brain.db` is a SQLite database with **FTS5 full-text search** over every markdown file Hamzaish cares about (this folder + `meta/`, `factory/playbooks/`, `products/*/`, `stack/`, plus root docs). It's **derived state** — generated by `ingest.ts`, never edited by hand, gitignored.

The `entities` and `edges` tables are stubs for Phase C (gbrain-style graph wiring). Empty until then.

## How to use it

### From the CLI

```bash
# refresh the index after writing new files
bun brain/ingest.ts                       # incremental (fast, ~50ms for 115 docs)
bun brain/ingest.ts --rebuild             # nuke + reingest everything
bun brain/ingest.ts --verbose             # list every file touched

# query
bun brain/ask.ts "what's the muakkil status"
bun brain/ask.ts --product muakkil "scribe demo plan"
bun brain/ask.ts --source brain/learnings "what surprised me"
bun brain/ask.ts --limit 4 --json "query"   # machine-readable
```

### From inside Claude Code

Use the `/brain-ask` command (`factory/commands/brain-ask.md`) — Claude shells out to `bun brain/ask.ts` and gets back markdown citations with paths it can then `Read`.

### When to re-ingest

- After writing a new learning, decision, or playbook entry
- After registering a new product or shipping a sprint
- Before opening a fresh Claude Code session if files have changed since the last run

Ingest is idempotent and change-detected (mtime + content hash). Re-running is free if nothing changed.

## What's NOT in the index

- `_archive/` — preserved old state, deliberately invisible to search
- `references/` — third-party repos, study material only, not factory knowledge
- `node_modules`, `.git`, `.next`, build outputs
- The reference repos have their own brains (gbrain has llms.txt, etc.)

## Schema

See [`schema.sql`](schema.sql). Three live tables, two stub tables, one audit log:

- `documents` — one row per ingested file (path, source tag, product scope, title, body, mtime, content hash)
- `docs_fts` — FTS5 mirror (kept in sync by triggers)
- `ingest_runs` — audit log of every ingest pass
- `entities` (stub, Phase C)
- `edges` (stub, Phase C)

## Roadmap

| Phase | Capability |
|---|---|
| **A.5 (now)** | FTS5 search over markdown; CLI + slash command |
| **B** | Use it on Muakkil's buildathon — capture what queries actually get asked |
| **C** | Add vector embeddings (Voyage/OpenAI); hybrid RRF (vector + BM25); entity extraction → `entities`/`edges` |
| **D** | Self-improvement hooks — automatic learning extraction at session end; eval harness wired |
| **C+** | Wrap gbrain as a sibling process if our minimal layer hits a ceiling |

## Why FTS5 first

For ~hundreds of personal-corpus markdown files, FTS5 is shockingly good:
- 50ms full reindex
- BM25 ranking out of the box
- Phrase queries, NEAR, prefix matching, snippets
- Zero external dependencies (built into SQLite)

Vectors win on semantic/conceptual queries. Until we have evidence those queries dominate, the simpler thing is correct. gbrain itself uses **vector + BM25 + RRF**, not vector alone — BM25 is foundational.
