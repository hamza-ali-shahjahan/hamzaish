# Learnings — 2026-07-03

## An unknown author on your own repo is usually attribution, not intrusion — but prove it, fast

**Context:** the repo's front page showed a commit by "noreply" (unfamiliar avatar, China) — an apparent intruder. Three read-only checks resolved it in minutes: `git log --format="%an <%ae>"` (the *name* was ours, the *email* a placeholder), `git reflog` (a plain `commit:` entry — created on this machine), and GitHub's activity API (pushed by the owner's own account). Root cause: a repo-local `user.email = noreply@users.noreply.github.com` override; GitHub maps that address to the real GitHub user "noreply". PR squash-merges had masked the bad config for 153 commits; the first direct push exposed it.

**Lesson:** GitHub attributes by email, and placeholder emails belong to *someone*. The scare has a mechanical diagnosis path (name→email → reflog → push actor) that should run before any intrusion response — and the prevention is layered, not documented: setup checks identity on every fresh clone (with the personalized fix printed, config never edited silently), auto-push hard-refuses placeholder identities, `/pr` pre-flights it. Also: masked failures surface at the *unusual* path (direct push), so the guard belongs on every path, not the common one.

**Where it's enforced:** `scripts/setup.ts` step 2 · `scripts/auto-commit.sh` (warn on commit, refuse push) · `/pr` pre-flight · `brain/anti-patterns/misattributed-commits-placeholder-email.md` (with the diagnosis checklist) · ledger #139 (✅). Repo-level: the require-PR ruleset's admin bypass removed — the unreviewed-direct-push channel is closed for everyone.

## A protection bypass for yourself is a protection with a hole shaped exactly like your own mistakes

**Context:** the "require PR + guards" ruleset was active the whole time — but the repo-admin role sat on its bypass list, so the misattributed commit walked straight onto main without PR, CI, or review. Every guard built this week (eval ratchet, retro gate, count sync) runs in PR CI; the bypass made all of them skippable by the person most likely to be moving fast.

**Lesson:** the owner's own sessions are the highest-volume committers and therefore the most likely source of process mistakes — exempting them inverts the protection. Removed the bypass; the owner now goes through the same `/pr` + guards as everyone. Emergency hatch stays available (ruleset edit via API), which is the right shape: possible, deliberate, and loud — never ambient.

**Where it's enforced:** ruleset 18167440 `bypass_actors: []` (verified); documented in the v2.5.4 changelog entry.
