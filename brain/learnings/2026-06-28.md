# 2026-06-28 — Learnings

## Recompose, don't regenerate, when the art is loved but the composition is broken

**Setting**: The README's Builder Mode cover cut the operator's illustration in half — but the *art* was fine; an overlay ("Fable 5") card sat on top of the lower body. "Recreate the banner" could have meant re-illustrate. It didn't.

**Lesson**: When a visual is wrong because of *composition* (crop, overlay, aspect ratio) and the underlying art is something the user values, **recompose from the original source — never re-illustrate.** Regeneration risks style-drift on the exact thing they like. Treat it as a layout problem, not an art problem.

**Technique that worked (deterministic, reviewable — no AI redraw)**:
- Composited the full original illustration into an SVG (`<image>` data-URI) and rasterized with `rsvg-convert` — byte-faithful art.
- **Sampled the art's own background color** from its corners (`#080b16`, via `ffmpeg` crop→rawvideo) and painted the canvas that color, then **feathered the art's left edge** into it with an alpha mask → the raster melts into the banner with no rectangular seam.
- Switched to a **half-height 1920×540 strip** (was 1080) so the README opening isn't a vertical wall — which also dropped the file 1.7MB → 573KB.

**The reproducibility miss it exposed**: the old banner shipped as a flattened PNG with **no source art and no build script**, so it couldn't be "uncropped" — the figure had to be rebuilt from the original file found *outside* the repo. **Commit the source + the build script, not just the export.** Now `docs/assets/hamza-meditation.png` + `scripts/build-banner.py` regenerate it, and the one file feeds both the README hero and `docs/builder-mode.md`.

**Decision**: standing default — fix asset *composition* by recomposing the committed source, and ship a build script for any generated asset. (Shipped as v1.29, PR #10.)

## Stage surgically when the working tree carries unrelated changes

**Setting**: While shipping the banner, the tree also held several **pre-existing unrelated edits** (README, start-here, a product config) plus untracked dirs from other work. `main` is branch-protected.

**Lesson**: with unrelated work present, **never `git add -A`.** Stage only the intended files by explicit path, then **verify `git show --stat` lists exactly those files** before pushing. Branch → PR → squash-merge (protected main); the unrelated changes stayed untouched and uncommitted, exactly as found.

**Decision**: explicit-path staging + a `git show --stat` pre-push check is the default whenever the tree isn't clean. Reinforces the `/pr` flow.

## "Hand-maintained facts drift" — the same wound the review and both ThousandWorlds repos hit independently

**Setting**: A full multi-agent **ultracode review** of Hamzaish ran this session (34 agents; report saved at `/private/tmp/claude-501/.../tasks/w1fz9qocp.output`). Its #1 verified finding: the "every count real" headline numbers have **drifted** (practices 128 vs 130 vs 133 on disk; playbooks 39 vs 38 vs 41; skills "17" omits two real skills; security checks "59" vs "80+" vs 65), and the path-portability hard rule is **breached and unenforced** — `products/copyright/product.config.json:9` ships a real `/Users/...` `code_path` and passes CI green.

**The pattern, seen three times in one session**:
- **Review** — counts are duplicated across `README.md`, `BEST-PRACTICES.md`, and `scripts/hero.ts` with nothing *deriving* them → drift.
- **ThousandWorlds contribution** — `n_folds` is `3` in code (`_run_model_config.py`) and prose, but `5` in every committed `config.json` artifact.
- **The emulator** — solved it the right way: caveats are **queryable data** (`observed[]`/`assumed[]`/`note` per planet in `src/lib/planets.ts`), so they can't fall out of sync with the values they annotate.

**Lesson**: any fact duplicated across code, prose, and a serialized artifact **will** drift. **Derive it from one source, or assert equality in a test/CI guard — don't hand-maintain copies.** Distilled into `brain/anti-patterns/hand-maintained-facts-drift.md`.

**Next move (from the review — opt-in sprint, not yet done)**: one `scripts/check-counts.ts` (derive every headline number from the filesystem, fail CI on drift) + a tracked-file `/Users/` path grep + a `code_path === null` assertion — closes count-drift, the `copyright` path leak, and doc-link rot in a single guard, which is exactly the repo's own "every rule that bit us earns a guard" ethos. Quick wins alongside: set `copyright` `code_path: null`, fix `pnpm`→`bun` at `factory/commands/hamzaish.md:58`, ship `code-paths.example.json` as `{}`, drop the 404 `ip-radar` showcase link.

## "It resolves" ≠ "it ships here" — verify provenance, and consolidate undeclared companions

**Setting**: Explaining the review's "front door references missing commands" finding, I (echoing the review) called `/full-cycle`, `/spec`, `/plan`, `/build` "Claude Code built-ins." The operator corrected me from memory — *he built `/full-cycle`.* He was right: those commands live in his own public repo `agent-skills`, installed globally at `~/.claude/commands/`, which is the only reason they resolve in a Hamzaish session. Not Anthropic's. The wrong claim had already reached a committed changelog line.

**The lesson**: a command resolving in your session tells you it's *installed somewhere*, not *where from*. "Not in this repo" does **not** imply "vendor built-in" — it could be your own separate repo on the global path, a companion project, or a plugin. Before stating provenance, find the defining file (`find . ~/.claude ~/Claude -iname '*<name>*'`; check its `git remote`). Distilled into `brain/anti-patterns/assuming-provenance-of-a-resolving-command.md`. This is the review-time face of [[hand-maintained-facts-drift]]: provenance is a fact, verified from source, not inferred from behavior.

**The fix that followed**: rather than just *declare* the dependency, we **consolidated** — folded the whole agent-skills engine (21 skills + 9 commands + 3 subagents) into Hamzaish so the front door is self-contained on a fresh clone and there's one repo to maintain (v1.31). The `check-counts` guard then caught every drifted count from the addition — the guard earning its keep on its first real change.

## Version numbers drift like any other hand-maintained fact — give them one home + a guard

**Setting**: Asked "are we at v1.31?", I found Hamzaish's version stated **four** different ways: `package.json` `1.10.0`, a `CLAUDE.md` line `v1.4`, the changelog `v1.31`, the last git tag `v1.2.0`. Nothing reconciled them — the exact [[hand-maintained-facts-drift]] wound, in the version number itself. "v1.31" wasn't even a release — it was the ~31st changelog *entry label*.

**The lesson**: a version is a fact, and facts duplicated across files drift. Untangle the two ideas that got merged: **changelog entries** (frequent, dated, iteration-labelled — notes) vs **releases** (deliberate semver git tags cut at milestones). Keep the version in **one** source (`package.json`), have docs *point* to it rather than restate it, and bump MAJOR only for a milestone — not by counting. Wrote the policy as `docs/versioning.md` (usable by forks too) and extended `check-counts.ts` to fail CI if `package.json` and `docs/versioning.md` disagree or `package.json` falls behind the latest tag. Cut **v2.0.0** for the self-contained-engine milestone.

**Decision**: from v2.0.0, Hamzaish versions by the policy in `docs/versioning.md`; the number is CI-guarded against drift, exactly like the headline counts.

## The build flow is goal-first and eval-gated: GOAL → SLICE → spec → build

**Setting**: The operator noted `/builder-mode` jumped straight to the spec, that "specs are a result of the right goal," and proposed a "slice" step that selects only features you can evaluate and end-to-end test.

**The change**: `/full-cycle` now pins the goal at kickoff (one measurable line, or `/write-a-goal` for ambitious targets), then a new SLICE gate (the `feature-slicing` skill) cuts the goal into vertical slices — each declares an eval + an end-to-end test, and a slice you can't prove isn't selected. The spec is written for the selected slices; their evals/e2e tests are the per-task TDD gate. `/goal` (the autonomous self-verifying loop) was shipped into the repo (it was global-only — same provenance gap as the consolidation) and elevated to a first-class mode.

**The lesson**: eval-driven development belongs at the *feature* level, not just the project level — the slice is where "the eval before the code" actually gates what gets built. The keeper rule: *if you can't name the eval and the end-to-end test for a slice, it doesn't get built yet.* And it composes with momentum — goal is one line by default, the slice gate is skippable, the full `/write-a-goal` forge is the escalation — clarity without a strategy toll.
