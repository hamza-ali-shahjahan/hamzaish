# 2026-06-20 — Learnings

## Rot is a stage, not an accident — `/tidy` makes cleanup a first-class move

**Setting**: the `check-assets` guard caught a real bug (committed docs linking to gitignored `meta/research/` files — fine on the author's Mac, 404 for every reader and in CI). The operator asked: make the guard better, make rot-removal a *core stage*, and make it work across 100+ repos.

### What we learned
- **"Passes locally, fails for everyone else" is a whole bug class, not one bug.** A link resolves on the author's disk because the target is *gitignored* (present locally, absent in the repo); a relative path is off by one `../` (works in one editor's resolver, 404s on GitHub); a case-mismatched path passes on macOS, dies on Linux CI. The author's machine is the *least* representative environment. The fix is to check against **what's committed** (`git ls-files`, `git check-ignore`), not what's on disk.
- **Gate ≠ tool.** A *gate* (`check-changelog`, `check-assets`) must be narrow, fast, deterministic, and block every push. A *cleanup tool* (`/tidy`) is broad, slow (network dep checks), report-first, and run on purpose at a milestone. Conflating them makes the gate flaky and the cleanup toothless. Keep them separate: the gate guards the diff; the tool sweeps the repo.
- **Report-first is the safe default for anything that touches unfamiliar repos.** `/tidy --all ~/Claude` swept 26 repos and ranked them by rot without editing a byte. Across 100+ repos you've never audited, *seeing* the extent first — then fixing with per-change confirmation — is the only sane posture. A tool that auto-edits repos it's never seen is a footgun.
- **Summary-first beats detail-dump.** The operator's instinct: show the extent per category (🔗 / 🔑 / 🗑 / 📦) before any details, then recommend, then ask. A wall of 200 findings is ignored; a 4-line summary + "what next?" gets acted on.

### What shipped
- **`scripts/tidy.ts` + `bun run tidy`** — portable rot scanner (any repo; `--all <dir>` for many). Four categories: broken/gitignored links, committed-secret patterns (review-grade), orphaned assets, npm-unresolvable deps. Report-first; `--fix` is opt-in + confirmed.
- **`/tidy` skill** — the cleanup *stage* (`factory/skills/tidy/SKILL.md`): run → summary → recommend → ask → fix-with-confirmation.
- **`check-assets` hardened** — now flags references that resolve *only* to a gitignored target, so the original class is caught locally, not just in CI.

### Status: green. Tidy is portable + multi-repo; ran clean across the operator's machine. Next: wire `/tidy` into the momentum router's "milestone / pre-launch" moment; add case-sensitivity + `--fix` to the engine.

## The factory already knew the bug — the miss was not consulting it (npm `bin` strip)

**Setting**: publishing the spun-out `rotscan` CLI to npm. `npm publish` warned `bin[rotscan] script name rotscan.ts was invalid and removed` — the published package would have shipped with **no `rotscan` command at all**. Caught because the publish then failed on the 2FA OTP, so nothing went live. Fixed by pointing `bin` at an extensionless `bin/rotscan` launcher; verified end-to-end (tarball → `npm install` → the `.bin` symlink runs).

### What we learned
- **I misdiagnosed the cause in the moment, then tested it.** I blamed the `.ts` extension. A 6-variant matrix proved otherwise: `./x.ts` `./x` `./x.mjs` all get stripped; `x.ts` `x` `x.mjs` all survive. **It's the leading `./`, not the extension.** Lesson: when you "know" the root cause under time pressure, a 2-minute matrix beats a confident guess — especially before you write the cause into a learning.
- **The knowledge already existed and still didn't save us.** [oss-publishing-checklist.md](../../factory/playbooks/launch-stage/oss-publishing-checklist.md) gotcha #2 is *exactly* this, with a pre-publish checklist item. It failed because rotscan was spun out fast and **nobody ran the checklist.** A doc that exists but isn't consulted has already failed once. The durable fix for "a human forgot to read the doc" is **make it mechanical**, not "write the doc again."
- **The checklist's own advised command wouldn't have caught it.** It says run `npm pack --dry-run` — but `npm pack` keeps the bin **silently**; only `npm publish --dry-run` runs the normalization that strips it. The verification step was pointed at the wrong command.
- **A raw `.ts` bin is a separate, real constraint** (not the strip cause): it only runs under `bun`; `npx`/node can't execute it. For a bun-only tool that's fine; for node reach, ship an extensionless/`.mjs` launcher. Worth stating so the next package doesn't conflate the two.

### What shipped
- **`scripts/check-npm-bin.ts`** — a mechanical preflight: flags any `./`-prefixed bin target (the strip), missing target files, raw `.ts` bins (bun-only note), and missing shebangs. Exit 1 on publish-breaking errors. Tested: PASS on fixed rotscan, FAIL on a `./cli.ts` fixture. This is the "turn the lesson into a guard" move — so the rule is enforced, not just written.
- **oss-publishing-checklist.md #2 amended** — empirical proof table, the `npm pack` vs `npm publish --dry-run` correction, the bun/`.ts` launcher distinction, and a pointer to the guard.

### Surprise (highest signal): the failure wasn't missing knowledge — it was knowledge sitting in a doc nobody opened. Every "we already had a playbook for this" miss is really a process gap that wants a guard. Pattern to watch: when spinning out anything new (a package, a repo), the relevant checklist must be *run*, ideally automatically, not remembered.
