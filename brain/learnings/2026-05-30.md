## Cross-product learning 1: Output validation for code-gen tools

**The trap**: When the product is a tool that emits code/config for a downstream toolchain, the natural test surface is the tool's own unit tests + the tool's own check command. Both can be 100% green while the output is broken.

**Concrete instance**: wp-to-astro 0.6.0 shipped with `slug: z.string()` in its emitted Astro content-collection schema. Astro reserves `slug` and rejects any schema that includes it (`ContentSchemaContainsSlugError`). All 138 unit tests passed. All golden-file diffs passed. `astro check` reported 0 errors. The bug was invisible until someone ran `astro dev` on the output — at which point every page 500'd.

**Why `astro check` lied**: It only parses `.astro` files. The migrated output had ZERO `.astro` files at the time of check (only `.mdx` content + a `.ts` schema). "Result (0 files): 0 errors" is what passing looks like when there's nothing to check.

**Rule for the factory**: When the product is a tool whose output is code, the pre-launch verification must include **running the output in a real consumer environment**, not just linting it. The deepest gate the consumer offers is the right gate.

- ✗ Wrong gate: `golden_file_diff(output, expected) && astro check`
- ✓ Right gate: `output → astro dev → curl every emitted entity → HTTP 200 across the board`

Translatable to other products:
- Codegen for an SDK → `npm install + tsc against the consumer's actual import path`
- SQL migration tool → apply to an ephemeral DB at the consumer's schema version
- API client generator → run an actual request against the documented API

**Where this is now encoded**: `agent-skills/references/output-validation-checklist.md` (separate repo I made today; not part of Hamzaish but useful as a reference). The pattern should also be added to `factory/mvp/security-reviewer/SKILL.md` or a new `factory/mvp/output-reviewer/SKILL.md` when next a product fits this shape.

## Cross-product learning 2: OSS publishing gotchas (npm / GitHub / 2FA)

Six concrete pieces of tuition I paid today that future products shouldn't:

| Gotcha | Symptom | Fix |
|---|---|---|
| GitHub email privacy blocks first push | `remote: error: GH007: Your push would publish a private email address` | Configure `git user.email` to your GitHub noreply (`<id>+<login>@users.noreply.github.com`) BEFORE the first commit. Get the ID via `gh api user --jq '"\(.id)+\(.login)@users.noreply.github.com"'`. If commits already exist, `git rebase --root --exec 'git commit --amend --reset-author --no-edit'` rewrites the chain. |
| npm silently strips `bin` paths with leading `./` | Published package has no CLI binary; `npx <package>` errors `command not found`. Only signal is a one-line warning in `npm publish` output. | In `package.json` use `"bin": { "name": "bin/file.mjs" }` NOT `"./bin/file.mjs"`. Run `npm pack --dry-run` and read every line — there are MORE silent edits beyond this one. |
| 2FA with hardware security key (no TOTP) blocks publish | npm publish returns `EOTP — open this URL in your browser to authenticate`. The URL gets censored to `***` in non-TTY output (bash captures, CI logs) so the auth flow can't be driven from a script. | Three options. Best: generate a granular access token at npmjs.com with "Bypass 2FA when publishing" checked (NPM_TOKEN env var, works in CI). Cheapest one-shot: run `npm publish --access public --auth-type=web` in YOUR terminal so the URL prints unmasked, click, security key, done. |
| `pnpm publish` doesn't pass `--auth-type=web` through cleanly | The web-auth flow never triggers; gets stuck on OTP wall | Use `npm publish` directly for the publish step (you can keep pnpm for everything else). |
| You can't republish the same version | npm returns 403 even if you delete the version | Bump the patch (`0.6.0` → `0.6.1`), update CHANGELOG with "regression fix" note, republish. `npm deprecate <pkg>@<broken>` adds a friendly warning to anyone who pinned the broken release. |
| Post-publish "fresh install" smoke test is NOT optional | "It was on the registry, I assumed it worked" → 6 hours later: real-world test reveals the package's emitted output crashes downstream | After publish: `mkdir /tmp/test && cd /tmp/test && npm init -y && npm install <pkg> && exercise the bin/entry point + run the FULL downstream pipeline`. If you skip this, ship a patch within the hour when reality intervenes. |

**Where this is now encoded**: `agent-skills/references/oss-publishing-checklist.md` (same separate repo). Should also be added to `factory/launch/release-engineer/SKILL.md` or equivalent when a product enters publish-to-registry phase.

## Cross-product learning 3: Validate before build (the rule we broke)

**The rule (from Hamzaish README §The discipline)**: "Don't build before you validate. 5 conversations with target-profile users before production code."

**What we did**: 6 build passes + a 0.6.1 patch shipped before a SINGLE conversation with a WordPress refugee. The validation phase is now playing catch-up.

**What it cost**: Hard to say. We have a shipped product but no demand evidence. The v0.7 polish punch list (4 items) is currently guesswork — we don't know which item matters most because no real user has told us. If the real user prioritization differs from our guess (likely), we've spent the wrong tuition.

**Why we did it anyway**: The build was concrete and tractable; the conversations are uncertain and require initiating contact. Building feels productive in the short run; validating feels like procrastination until later when it turns out to have been the load-bearing step.

**The fix for next time**: When Hamzaish scaffolds a new product, the FIRST gate before any code is the validation conversations. Not after. Not in parallel. Before. The factory's `/scaffold` and `/validate` commands exist for this — use them, don't skip them.

**Recovery for ai-native-cms**: Sprint `validation` — 5 WP-refugee migrations done for free in exchange for feedback + price signals before any further build or HN launch. Exit criteria in `products/ai-native-cms/status.md`.
