# 2026-06-04 — Learnings

## Phase 0 — IP Radar: data-source pivot under a KYC wall + a local-index cost architecture

**Setting**: IP Radar (the US copyright + patent agent, `products/copyright`) needed live USPTO patent data. The intended source (USPTO Open Data Portal) turned out to gate its API key behind ID.me identity verification — discovered mid-build, after the integration was written. Pivoted to Google Patents BigQuery in the same session, then built a local pgvector patent index on top of it to kill per-query cost. Seeded 27,847 patents end-to-end. Shipped, live in prod.

These are the **reusable patterns** — for any future product that depends on a metered/gated external data source.

## Cross-product learning 1: Check a data source's auth/KYC model BEFORE committing to it

**The trap**: You pick a data source on coverage and price, build the integration, then discover at credential-issue time that it requires identity verification (ID.me, SSN, KYC, a US entity, a particular phone/document). That can be a multi-day or hard blocker — discovered after the integration code is already written.

**Concrete instance**: USPTO ODP gates its API key behind a MyUSPTO account linked to ID.me, whose verification routes through a video call and an SSN step. Hours were spent on the ID.me flow before pivoting.

**Rule for the factory**: When `/scaffold` or `/validate` picks an external data/API dependency, vet the signup path in the FIRST hour:
- Email-only signup? ✅ safe
- Requires ID.me / Login.gov / SSN / US entity / specific documents? ⚠ flag immediately and find a fallback before writing integration code.
- Always identify a second source up front. Here the fallback (Google Patents BigQuery) was strictly *better* — no identity wall, commercial-use OK, 100% coverage, free tier. The wall forced a better architecture.

**Translatable**: any gov/enterprise data API (IRS, SEC EDGAR bulk vs. authenticated, Stripe in some regions, Plaid, KYC'd fintech sandboxes).

## Cross-product learning 2: Scan-billed sources bill on COLUMNS SCANNED, not rows returned — `LIMIT` does not bound cost

**The trap**: You reach for `LIMIT 3` to make a "cheap test query" against a columnar/scan-priced warehouse (BigQuery, Athena, Snowflake). The bill is unchanged, because the engine evaluates the SELECT expressions across every row matching the WHERE *before* applying the LIMIT.

**Concrete instance**: A BigQuery seed query against `patents-public-data.patents.publications` with `LIMIT 3` demanded **370 GB billed** — because it selected the `claims_localized` column (the largest field in the table) and BigQuery scanned that column for every row in the partition window before limiting. Removing claims (seeding on title + abstract only) dropped it under the cap instantly.

**Rule for the factory**: For any scan-billed source:
- Select only the columns you actually need. The biggest text/blob column is usually the cost driver — exclude it unless essential.
- Filter on the partition/clustering key (here `publication_date`) to prune partitions — that DOES reduce bytes scanned.
- Set a hard `maximumBytesBilled` cap as a runaway backstop.
- Never reason about cost from the row LIMIT.

## Cross-product learning 3: The local-first index pattern — pay once to seed, search free forever

**The pattern**: When a product hits a metered external API for *search* on every user query, the per-query cost scales with usage and becomes a pricing-model risk. Fix: seed a **local pgvector + tsvector index** once from the expensive source, run user search against the local index (free), and keep the expensive source as (a) a fallback when the local index is sparse and (b) a periodic refresh job.

**Concrete instance**: Patent search went from ~$1.50 per cold BigQuery scan to $0 per query. Seed once (`pnpm ingest:patents-bq`, ~$1-2 one-time Voyage embed, BigQuery within free tier) → 27,847 patents in `patent_index` → `search_patents` is local-first via RRF hybrid (vec 0.7 / fts 0.3); BigQuery only fires when <5 local hits. A weekly cron tops up a recent 90-day window (cheap — partition-pruned). This mirrors the existing `case_law_index`.

**Rule for the factory**: any product doing search over a metered API should default to this shape. Per-query metered cost is a silent pricing killer; a seed-once local index converts it to a one-time + cheap-refresh cost. Reusable across legal data, patent data, product catalogs, any "search a big external corpus" feature.

**Where this should be encoded**: a `factory/playbooks/` entry — "Local-first index for metered search APIs" — with the pgvector + RRF + seed/refresh/fallback template.

## Cross-product learning 4: Provider-SDK wrappers must enforce the provider's batch limits INTERNALLY

**The trap**: A thin `embed()` / `client.batch()` wrapper passes the caller's array straight through. It works in dev (small arrays) and 400s in production ingest (large arrays) when the provider's per-request batch cap is exceeded — failing at the worst time, mid-bulk-job.

**Concrete instance**: Voyage caps embeddings at 1000 docs/request (plus a token cap). The seed sent 2,500 at once → `batch size limit is 1000`. Fix: `embed()` now chunks internally by BOTH document count (128) and a character budget (~90K tokens), preserving order. Every caller — bulk seed, weekly refresh, case-law seed — is now safe with zero call-site changes.

**Rule for the factory**: provider wrappers own the provider's constraints (batch size, token limits, rate limits, retries). Callers should never have to know them. Bake the limit into the wrapper the first time you write it.

## Cross-product learning 5: A per-item try/catch in a fan-out loop can disguise PARTIAL failure as success

**The trap**: Bulk loops wrap each item in `try/catch { log; continue }` so one bad item doesn't sink the run — good. But then the run prints "Done. Total: N" and looks complete, while one item silently produced zero output. The summary line hides the gap.

**Concrete instance**: The patent seed loops over 12 CPC subclasses. The G06F21 (security) pass errored, the catch logged-and-continued, and the run finished "Done. Total patents indexed: 27,847" — looking complete. Only a later targeted verification revealed G06F21 had **0 rows**. The success summary masked a missing subclass.

**Rule for the factory** (ties directly to the README discipline "verification is non-negotiable / no silent caps): bulk loops with per-item catch must **track and report which items failed**, not just log-and-continue. The final summary should read `Done: 11/12 subclasses (G06F21 FAILED: <reason>)`, never a bare total. And post-ingest verification should check per-category presence, not just a grand total. "Done" must never be able to hide a partial failure.

## Cross-product learning 6: Idempotent bulk ingest (upsert on natural key) makes long jobs safe to re-run

**The win** (the one thing that went right): the seed used `onConflictDoUpdate` on `patent_no`, so every partial failure was recoverable by just re-running — the batch-limit failure and the G06F21 gap both top up cleanly on re-run, no dedup logic, no manual cleanup. Design bulk ingests as idempotent upserts from line one; it converts "the job died at 70%" from a disaster into a shrug.
