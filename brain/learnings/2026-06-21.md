# 2026-06-21 — Learnings

## Wiring a local vision model into the builder: download → install → enable → test

**Setting**: the Local LLM builder cloned websites *structurally* only (text digest → coder
rebuild → structural score), so visual fidelity plateaued ~30–46%. The Capabilities modal even
named "a vision model" as the lever, but nothing consumed it — "Vision-critique clone loop" was
marked *Soon*. Two operator asks, back to back: (1) make the modal's "install the vision model"
actually clickable and make it obvious what's *live*; (2) build the loop so the model is genuinely
leveraged for cloning. Both shipped + dogfooded on the live builder.

### What we learned

- **The headline trap — a local vision model's default context (4096) overflows on two images.**
  Each screenshot ≈ 1,900 vision tokens; two images + a real prompt ≈ 4,168 > 4,096 → Ollama
  `exceed_context_size_error` (HTTP 400), *before any inference*. The fix is one line:
  `options.num_ctx: 8192` on the `/api/chat` call. Captured on its own:
  [vision-model-default-context-overflow](../anti-patterns/vision-model-default-context-overflow.md).
- **A swallowed error body cost the most time.** The endpoint returned `str(HTTPError)` = "Bad
  Request"; the real cause (exact token counts, `n_ctx:4096`) sat unread in `e.read()`. Always
  surface the upstream error body. And a *short-prompt* smoke test passing made it look like the
  endpoint — not the payload size — was at fault (same images, 30-token vs 250-token prompt = pass
  vs fail). Test with the real prompt + real image dims.
- **Frontend and backend have opposite reload semantics.** The builder serves `index.html`
  `no-store` → frontend edits go live on reload. The Python agent server loads code at startup →
  backend edits need a *restart*. Easy to "test" stale backend code without noticing.
- **Restart the server with the SAME interpreter/venv or you silently downgrade.** The agent server
  ran from a dedicated `.agent-venv` (where Playwright lives). A naive `python3 agent-server.py`
  would have dropped the screenshot backend Playwright→Chrome (lower clone fidelity) with *no
  error*. The tell was `__PYVENV_LAUNCHER__` in the running process's env; verify
  `screenshot_backend` in `/api/agent/ping` after restart.
- **You can install a model straight from the browser.** Ollama's `POST /api/pull` streams NDJSON
  progress, and the page already POSTs `/api/chat` so CORS is open — no backend needed; re-poll
  `/api/tags` to flip the row to "Installed." **But only offer it for models used as-is**: the
  vision model pulls cleanly; the coder tiers get re-imported under custom context/lean alias
  Modelfiles, so a raw pull would be a half-install.
- **"Live vs installed" can't ride on lookalike emojis.** ✅ active and 🟢 available read
  identically — a user can't tell what they can use *right now*. Worded, colour-coded pills
  (Installed / Not installed / Needs more RAM / Soon) fixed it.
- **RAM reality for local multimodal.** qwen2.5vl:7b is ~6 GB but wants ~24 GB to run *alongside*
  the 14B coder; on 24 GB Ollama swaps models each round → clones get slow. Bound the rounds and
  say so up front.

### What shipped

- **Click-to-install in Capabilities** — a real Install button that streams Ollama's pull with a
  live progress bar and auto-flips to ✓ Installed; plus worded status pills across models +
  capabilities.
- **Vision-critique clone loop** — backend `/api/agent/visioncritique` (screenshot clone+target →
  vision model → `{visual_score, summary, fixes}`, with `num_ctx:8192` + surfaced error bodies);
  frontend runs the loop (critique → feed fixes to coder → re-render → re-score), ending on a
  critique so the shown score matches the app. The capability flips Soon→Live when the model + the
  agent server are present.
- Verified end-to-end on the live builder (real critique with score + actionable fixes), 34/34
  tests green, baked into both installers. The same learnings were written into the Local LLM repo
  itself (`docs/vision-model.md` + a README pointer) so contributors and users get them too.

### Surprise (highest signal): the slowest bug to find was the cheapest to fix. A one-line
`num_ctx` overflow masqueraded as an image-format / API-shape problem for an hour — purely because
the 400 said "Bad Request" instead of its own JSON body. Two cheap habits collapse this whole loop:
**read the error body**, and **smoke-test with the real prompt + image sizes**, not a convenient
miniature that passes for the wrong reason.

## A richer input only helps if you compress it, force it, and give it room — the "headroom" trio

**Setting**: pushing local website-clone fidelity. The render-based inspect captures a *rich* digest
(every computed colour, font-size, radius, shadow, spacing value). We'd been feeding that to a local
14B coder and getting ~64% structural — gated almost entirely by **fonts (25%) and design tokens
(0%)**. The operator pointed at [headroom](https://github.com/chopratejas/headroom) and asked: adopt
the token-optimisation spirit natively. Shipped as Local LLM `v1.17.0`.

### What we learned
- **Richer input is necessary but nowhere near sufficient.** We proved render-based inspect captures
  the real tokens (disrupt.com's tan accent `rgb(243,213,186)` that the stdlib parse missed). Yet the
  clone still scored tokens **0%** — the 14B was handed an exhaustive dump and *dropped almost all of
  it*. A better input the model can't consume changes nothing. (Sharpens the earlier v1.14.0 note.)
- **The fix is a trio, and all three move together:**
  1. **Compress at the source** (the "headroom" spirit) — dedup, cap, round (`18.52px → 18px`), and
     **drop low-signal arrays entirely** (spacing, here). disrupt's ~30 token values → ~12. Leaner
     context = less for the model to drop, and rounding even makes the clone's round values *match*
     the original's. Crucially, compress where the digest is *built*, so every downstream
     tool-output AND the spec inherits it — not at each call site.
  2. **Force what survives.** Once it's compact, state the few high-signal tokens as **exact
     must-implement values** ("corner radius: use exactly 8px / 12px"), not a vague "polish" note. A
     small model obeys a short list of hard constraints; it ignores a long list of soft ones.
  3. **Give it room.** An 8k context truncated *both* the rich spec and the full HTML output → thin,
     low-scoring first passes. Bumping clone calls to 16k (`options.num_ctx`, per-request — no model
     re-import) let the input and the output coexist.
- **Single-shot local-model output is high-variance — never benchmark on one run.** The same coder +
  spec gave a 67-line/19% clone and a 169-line/63% clone on consecutive tries. Only the **full loop**
  (build → score → refine → vision-critique → keep-best) produces a trustworthy number; a one-shot
  comparison "proves" nothing. Drive the whole loop, or you're measuring noise.

### What shipped
- **`_compress_tokens`** (agent server): dedup + cap + round-px + drop spacing on the token arrays;
  the fidelity scorer reads the same compressed tokens.
- **Token-forcing** in the clone spec; **16k clone context** on all three coder calls (gen/refine/
  vision-fix); **vision rounds 2 → 3** (4 in Goal). 34/34 tests green (incl. token-capture +
  scoring), shipped `v1.17.0`.

### Surprise (highest signal): the dependency-free win beat the obvious one. The instinct is "bigger
model" — but on 24 GB a 32B coder just swaps with the vision model and crawls. The actual lever was
**making the input the existing 14B could already consume**: compress it, force it, give it room. A
better *prompt-shape* outran a bigger *model* for the same money (zero).
