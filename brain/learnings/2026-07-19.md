# 2026-07-19

## A 102-page survey can be a two-item diff

Read the code-as-agent-harness survey (arXiv:2605.18747) against the factory instead of absorbing it wholesale: most of its taxonomy mapped onto things that already existed (guards = code as boundary, evals = deterministic sensors, wip(auto) = stateful traces), leaving exactly two real gaps — session traces and the lesson→check ladder. The move that made this cheap: read the paper *holding the repo in mind*, listing "have / don't have" per section, then build only the don't-haves. Surveys are checklists for mature systems, blueprints only for greenfield ones.

## Project hooks go live on write, no restart needed

Creating `.claude/settings.json` with PostToolUse hooks mid-session started capturing that same session's tool calls immediately — the build verified itself against the real harness before its fixtures-based tests were even relevant as the primary evidence. When adding observability, check whether the observer can watch its own installation: it's the cheapest possible end-to-end test and it exercises the real payload schema, not your guess at it.

## Failure counts must be declared floors when detection is heuristic

Hook payloads carry no bash exit code, and stderr-presence is not failure (git, curl chat on stderr constantly). The honest design: flip `ok=false` only on explicit error signals (`is_error`, `interrupted`, an `error` field), and label every aggregate as a floor in both the report footer and the script header. The alternative — inferring failure from stderr — would have made the report confidently wrong, which is worse than modestly incomplete (same shape as the telemetry connectors' "no key never gets invented numbers" rule).
