# 2026-06-16 — Learnings

Session: Patently Legal (IP Radar) — index-deepening + cost/latency hardening. Three durable engineering lessons, all from real money or real misdiagnosis on a live product. Sources: CourtListener v4 search API, Google Patents BigQuery, Voyage `voyage-law-2` + Neon pgvector.

## A dedup-on-insert seed silently froze the corpus at 1,054 — and it looked like a rate-limit problem for hours

**Setting**: deepening `case_law_index` from CourtListener toward 30k. The seed paged `page=1,2,3…120` per query, logged `page N: 20 candidates → embedding` for every page, and *looked* perfectly healthy. The table count never moved off **1,054**.

### Root cause (a two-part trap)
- **CourtListener v4 search is CURSOR-paginated.** Under `order_by=score desc` the `page=N` param is silently *ignored* — every page returns the **same first 20 results**. The real pagination token is the `next` field (a complete `?cursor=…` URL) you must fetch verbatim.
- **`onConflictDoNothing` turned a broken loop into a silent no-op.** Pages 2…120 were 100% duplicates of page 1, so the idempotent insert dropped them all without error. The seed reported "running fine," embeddings were billed, logs scrolled — and net-new rows were **zero**. We could have run 10,000 pages and never grown past ~1,054.

### Why it cost hours
The symptom (count stuck) co-occurred with genuine 429s, so the first hypothesis was "CourtListener rate limit." We tuned the throttle, added backoff, lowered the request rate — all irrelevant. The actual proof took 30 seconds once asked the right question: **fetch page 1 and page 2, compare the first cluster IDs.** They were identical → pagination was the bug, not the rate limit.

### Guardrails (every paginated ingest, no exceptions)
1. **Instrument NET-NEW rows per batch, not rows attempted.** `ingested += chunk.length` after a dedup insert is a lie — it counts duplicates as progress. Log `count(*)` deltas, or use `INSERT … RETURNING` and count returned rows. A flat total under a climbing "pages processed" is the tell.
2. **Before trusting any page loop, assert pages return DISTINCT keys.** Two-call smoke test: fetch page 1 and page 2, diff the IDs. If they overlap fully, the pagination param is being ignored — you're almost certainly on a cursor API. Follow `next`/`cursor`, never assume `page=N` works.
3. **Don't let an idempotent write mask a broken producer.** `onConflictDoNothing`/upsert is correct for resumability but it *hides* total upstream failure as "healthy." Pair every dedup sink with a freshness signal (net-new count, or a periodic `select count(*)`).

→ promoted to anti-pattern: [`silent-dedup-masks-broken-pagination`](../anti-patterns/silent-dedup-masks-broken-pagination.md)

## Never put a pay-per-byte scan on a per-user code path — it was $10 per clearance run

**Setting**: each IP-clearance run cost **~$10.77**. Not viable at any sane price point.

### Root cause
`claim_similarity` fetched patent claims live from Google Patents BigQuery (`claims_localized`) **per patent, per run** — each scan touched ~370 GB, and BigQuery bills per byte scanned. A handful of patents × a 370 GB scan each = ~$10 of BigQuery on a single user request. The LLM and embedding costs were noise by comparison; the dollars were entirely in the data-source scan.

### The fix — the "cost-killer seed"
Pre-compute the expensive thing **once, offline**, into a **local index**, then serve from local at request time. We already had title+abstract embeddings in `patent_index` (halfvec); switched claim similarity to cosine over those local vectors. Per-run BigQuery cost: **~$10.77 → ~$0.10**. The one-time BigQuery seed scan stays within the free 1 TB/month tier; user-facing requests touch zero BigQuery.

### Guardrails
1. **No metered external scan (BigQuery, Athena, any pay-per-byte engine) in a synchronous user path.** If a request can trigger it, an abusive or just-popular day is an unbounded bill. Move it behind an offline ingest into Postgres/pgvector/SQLite and serve locally.
2. **Profile where the dollars *actually* are before optimizing.** Instinct said "it's the LLM"; the truth was a data scan costing 100× the model call. Attribute cost per surface (we have `runWithCostSink`) before cutting anything.
3. **One-time ingest cost ≠ per-request cost — keep them in separate budgets.** A $1-2 corpus build is fine; the same scan per request is fatal. The asymmetry is the whole point of a local index.

## Voyage embeddings were never the cost — and `halfvec` doubled what a free-tier DB can hold

**Setting**: corpus-scaling on free-tier Neon (512 MB hard cap), `voyage-law-2` (1024-dim) embeddings.

### What we learned
- **Voyage embeddings are cheap — ~$1-2 one-time for the *entire* corpus.** When the per-run bill was $10, the reflex was to suspect embeddings; they were a rounding error. The expensive thing was the BigQuery scan that *fed* the embeddings, not the embedding API. Don't blame the AI vendor before measuring.
- **`halfvec` (fp16) halves vector storage with negligible recall loss.** Switching pgvector columns from `vector(1024)` to `halfvec(1024)` ~doubled the corpus a 512 MB free tier holds. Migration order matters: **DROP the HNSW index first, then `ALTER TYPE`** (the index pins the column type). For our scale, exact KNN over halfvec was fast enough that we dropped HNSW entirely — simpler, smaller, no index to keep consistent.
- **Neon's HTTP driver chokes on large vector insert batches — chunk at ~5 rows.** Bigger batches fail opaquely; a `chunkSize = 5` loop is the reliable shape.
- **Free-tier Neon silently fails inserts once full.** No loud error — writes just stop landing. Treat DB size as a hard ceiling and watch `pg_database_size` as a first-class metric during any bulk ingest (this recurred enough to be a standing constraint).

### Guardrails
1. **Measure vendor cost before assuming it.** "It's probably the embeddings/LLM" is usually wrong; meter it.
2. **For embeddings on a storage-constrained DB, default to `halfvec`.** Drop the index → alter type → (re)build or drop. Half the bytes, ~same recall.
3. **Chunk vector inserts (≈5) and monitor DB size as a ceiling** on any free/oversubscribed Postgres.

## Status: green — all three live on Patently Legal. Cursor pagination fixed + smoke-tested (60 distinct IDs / 3 pages); per-run BigQuery $10.77 → $0.10; halfvec migrations shipped on patent + case indexes. Website + repo now quote rounded-DOWN floors (50,000+ patents / 1,000+ opinions).
