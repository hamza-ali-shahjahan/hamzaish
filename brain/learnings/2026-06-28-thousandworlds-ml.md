# 2026-06-28 — ThousandWorlds ML learnings (benchmark contribution + client-side emulator)

Durable lessons distilled (read-only) from two sibling projects in the ThousandWorlds family. Many of these are ML-specific, but several generalize to *any* build — they're the kind of discipline the factory should carry. Evidence cites the source artifact (file or commit), never a conversation.

- **ThousandWorlds** (`~/Claude/ThousandWorlds`) — an ML *research contribution*: a `PCA-GBT` baseline added to the upstream exoplanet-climate benchmark (PCA compression → one gradient-boosted-tree regressor per latent mode). Not a web app.
- **The emulator** (`~/thousandworlds-emulator`) — a React/Vite/TypeScript single-page app that runs that emulator **fully client-side** via ONNX Runtime WASM: 8 planetary inputs → a 32×64 temperature field in <1ms.

---

## A. Benchmark / ML contribution lessons (ThousandWorlds PCA-GBT)

- **Pick the model from the data's structure, not the trend.** The benchmark's headline was "deep learning doesn't beat Gaussian Processes." Rather than a bigger net, the contribution diagnosed *why* — climates flip between regimes (snowball / temperate / runaway) at sharp thresholds — and chose axis-aligned boosted trees *because* trees split exactly at thresholds where smooth models blur them. Beat every DL baseline on data-rich subsets at CPU cost. — *`code/CONTRIBUTION_pca_gbt.md` ("Why trees"); commit `172ad7c`* — **ml-eval**
- **Tune hyperparameters with CV on training folds only; take the honest number over the vanity one.** v1 used a hand-set config; the revision replaced it with a per-subset cross-validation sweep that sees only training folds, and explicitly accepted "honestly cross-validated" scores even where that costs a little vs a test-tuned config. Defensibility > a prettier number. — *`code/CONTRIBUTION_pca_gbt.md` ("Limitations / honest notes"); commit `9ca634a`* — **ml-eval**
- **Make every reported number replayable from a checked-in config.** The sweep writes chosen values, full grids, AND per-fold scores into each subset's `config.json`; `--config` replays the exact run and skips the sweep. Metrics read from committed JSON, not a one-off run — auditable months later. — *`code/results/models/multi-partial/pca_gbt/config.json`; commit `9ca634a`* — **reproducibility**
- **Own your weakness — a labeled win/lose boundary beats an all-green table.** The writeup leads with where trees *lose* (smooth radiative fields, where they "staircase" and GPs win) and that a bespoke GP is still SOTA overall, explained mechanistically. Owning the failure mode made the claimed wins far more credible. — *`code/CONTRIBUTION_pca_gbt.md` ("Limitations"); `code/README.md` ("Honest caveat")* — **ml-eval / communication**
- **Swap one component, leave the rest byte-for-byte.** PCA-GBT changed only the latent-score regressor; loading, PPCA, decoding, scoring are identical to the existing baselines — so the result can't be attributed to an incidental difference and the diff is reviewable. — *`code/CLAUDE.md`; `code/thousandworlds/models/pca_gbt.py`* — **technical-pattern**
- **Prune un-validated optionality before review.** The revision deleted two untested tree backends and a `--gbt-tree-backend` switch, leaving one well-tested engine. Optionality you haven't validated is liability, not flexibility. — *commit `9ca634a` ("drop non-boosting tree backends")* — **anti-pattern**
- **Parallelize the embarrassingly-parallel work, but pin each worker to one thread.** One regressor per latent component via joblib process-parallelism, each tree fit wrapped in `threadpool_limits(limits=1)` so inner BLAS/OpenMP threads don't oversubscribe the cores joblib already uses. Naive `n_jobs=-1` without capping inner threads = thread-explosion. — *`code/thousandworlds/models/pca_gbt.py` (`_fit_one_hgbt`)* — **technical-pattern**
- **Guard small-data edge cases that silently break library defaults.** HistGBT early-stopping carves an internal validation split that destabilizes on tiny sets, so early stopping is disabled when `n < 40` and CV picks shallow trees for the ~206-world subset vs deep for the ~1,600-world ones. A default that's fine at 1,500 rows misbehaves at 200. — *`code/thousandworlds/models/pca_gbt.py` (`n >= 40` guard)* — **technical-pattern**
- **Ship the contribution doc in two lanes: plain-English first, rigorous second.** One anchored document with a "Plain English" section and a "For scientists" section — the finding lands for a skimming maintainer and a learning newcomer without dumbing down the defensible version. — *`code/CONTRIBUTION_pca_gbt.md`; commit `9ca634a`* — **process**
- **A "real" benchmark entry is registry + tables + tests + docs, not one `.py`.** Registered the method in `PUBLIC_METHODS` (so it appears in auto-generated tables), added CLI flags + config resolver + round-trip, regenerated all result tables, added smoke/resolver/CV-replay tests, updated the rerun registry and README. The integration points *are* the deliverable. — *commit `9ca634a`; `code/tests/test_models.py`* — **process**
- **Know the platform's commit-hygiene gotchas before contributing.** Written down per-project: GitHub blocks pushes carrying a real email (use the `…@users.noreply.github.com` form), and `tw.download_dataset()` rewrites a tracked `dataset/README.md` that then dirties the PR diff unless reverted. One-time-painful, always-painful-again — record them. — *`code/CLAUDE.md` ("Gotchas learned the hard way")* — **process**
- **Committed artifacts drift from code/prose (the same wound as the Hamzaish count-drift).** `n_folds` is `3` in code default + every doc, but `5` in all three committed `config.json` files. Derive the artifact from the code, or assert their equality in a test — don't hand-maintain three copies. See `brain/anti-patterns/hand-maintained-facts-drift.md`. — *`_run_model_config.py:40,50` vs the `config.json` files* — **anti-pattern**

## B. Client-side inference + honesty-engineering lessons (the emulator)

- **Ship per-input preprocessing as a frozen artifact, not code.** Different physical inputs need different transforms before standardization — linear for bounded quantities, `log` for multiplicative ranges, `arcsinh(x/s)` for signed heavy-tailed trace gases spanning many orders incl. zero. The strategy + mean + std + kwargs ship as a per-input JSON so inference reproduces training preprocessing exactly. — *`public/emulator/input_transform_surface_temperature.json`; `src/lib/emulator.ts` `preprocessInput()`* — **ml-modeling**
- **Emit an explicit feature-order + decode contract as metadata.** A deployable model is weights *plus* `onnx_feature_order`, `design_row_order`, units, grid shape, and the literal decode formula (`grid = z @ G_lat.T + h @ G_trend.T + g_bias; reshape(32,64); Kelvin`). That contract — not tribal knowledge — is what let a JS port match Python `predict()` to ~5e-4 K. — *`public/emulator/emulator_meta.json`* — **reproducibility**
- **Verify a re-implementation against the source model with a hard numeric tolerance.** Don't eyeball a port to a new language/runtime — assert agreement to an explicit tolerance and record the number in the code ("verified to ~5e-4 K against the benchmark's `predict()`"). — *`src/lib/emulator.ts:8-9`* — **ml-eval**
- **A linear decoder is a feature: it ports to ~10 lines and runs sub-ms client-side.** Compress to a 16-dim latent, GBT per mode, then decode the field with a pure linear map — the architecture choice directly bought browser deployability and cross-language verifiability. — *`src/lib/emulator.ts:147-161`* — **ml-modeling**
- **Carry the simulator identity as a model input — different simulators genuinely disagree.** When training data pools multiple GCMs of the same system, a 5-way GCM one-hot belongs in the feature vector; selecting one at inference surfaces honest model-to-model spread on the *same* world. (Simulation-choice spread — never real-vs-observed.) — *`emulator_meta.json` (`n_sim_types:5`)* — **ml-modeling**
- **Track the in-distribution envelope; flag extrapolation rather than block it.** Persist training-data ranges + a tighter "densely simulated" sub-range; out-of-range inputs are still predicted but **labeled** "rough extrapolation," because the model is rougher where the training set was sparse even inside the nominal envelope. — *`src/lib/constants.ts:22-31`; `emulator.ts:163-167`* — **ml-eval**
- **Compute sensitivity at swept endpoints, never by scaling a local gradient.** For "what drives the output," sweep each input across its well-simulated range and re-run the model at both ends — bounded, and it saturates where the true response flattens (a steep local slope would blow up). Cost was fixed and tiny. — *`src/lib/sensitivity.ts`* — **ml-eval**
- **Re-verify the premise of an inherited caveat before carrying it forward — caution can ossify into a wrong caveat.** Earlier, two inputs (rotation, CH₄) were *filtered out* of the tornado as "units still being calibrated," on the principle that a footnote doesn't stop a bad signal from topping a ranking. **Correction (later session):** a ground-truth probe showed their input transforms (`log`, `arcsinh(x/s)`) were in fact *bit-exact* with the benchmark — never mis-calibrated — so they were un-hidden and rank cleanly. The durable lesson flips: re-test the premise, don't inherit the caution. (The footnote-vs-filter principle still holds *when* the signal is genuinely bad — here it wasn't.) — *`src/lib/emulator.ts:76-91`; commit `cbe19e6` (un-hidden)* — **process / ml-eval**
- **Centralize and inline-document every unit conversion — it's a top defect source.** The model trains in dataset units (m, Pa, mole fraction) while the UI speaks Earth-radii/bar/percent; all conversions live in one function with the source unit commented per line, so "calibration pending" is a known contained risk, not a silent 100× error. — *`src/lib/emulator.ts:76-89`* — **data**
- **Self-host the inference runtime — no CDN in the model's critical path.** The ONNX WASM runtime is copied into the build and served locally (`numThreads=1`, explicit `wasmPaths`) so the model runs offline, deterministic, and CSP-friendly. Owning the runtime bytes removes a live external dependency. — *`vite.config.ts` (`viteStaticCopy`); `emulator.ts:106-108`* — **deploy-launch**
- **Structure caveats as queryable data so they can't drift out of sync.** Real-planet entries carry `observed[]` vs `assumed[]` arrays (which params are measured vs Earth-like stand-ins), out-of-range derived values are clamped *and* flagged in a `note`, and even Earth is marked a "slight extrapolation." Caveats-as-data beat caveats-as-prose — they move with the values. (Same root pattern as `hand-maintained-facts-drift`.) — *`src/lib/planets.ts:4-10,32-40`* — **process / honesty**
- **Match the audience's expected visualization, and pick palettes that don't hide structure.** Reproduced the paper's continuous `RdYlBu_r` (a banded palette "flattens a temperate world to one green"); A/B diffs use a lightness-symmetric diverging map so neither warming nor cooling reads louder. Visualization fidelity is part of being faithful to the model. — *`src/lib/colormap.ts`; `src/lib/compareDiff.ts`* — **ml-eval / process**
- **Born-protected repo hygiene.** The `.gitignore` ships the "never commit conversations/transcripts/strategy notes" block + a strict `.env.*` net (with `!.env.example`) from line one, and documents the `env-unset → accounts-off` failure mode. Scaffold the protective net before the first commit. (Reinforces `brain/learnings/2026-06-26-no-conversations-in-repo.md`; this is exactly what the Hamzaish product starter now ships.) — *`.gitignore`; `.env.example`* — **process / deploy-launch**

### Added later (multi-field decode, level plots, uncertainty)

- **Verify a field's decode→physical map against ground truth, per field — a wrong inverse looks plausible, not broken.** The ASR map shipped reading ~0 W·m⁻² because that field is stored *divided by stellar flux*; nothing looked wrong until decoded training worlds were diffed against the ground-truth grids. Each output hid a different inverse — identity (temperature/winds), `× F_star` (ASR/OLR), `exp` (humidity), `sigmoid+clip` (cloud). Decode a few known worlds and compare to truth *before* wiring the UI. — *`scripts/export_fields.py`; `src/lib/emulator.ts` (`predictField`, `× flux`); commit `2368275`* — **ml-eval / anti-pattern**
- **A shared latent + per-field linear decode means a new output field costs only its decode matrices — no retraining.** All 53 benchmark fields share the same 16-dim GBT/PPCA latent, so ASR/OLR and then all nine level-field plots were added by exporting per-field `(G_lat, G_trend, g_bias)` and reusing the *same* ONNX. Look for this latent bottleneck before assuming a new model is needed. — *`scripts/export_fields.py`, `scripts/export_levelfields.py`* — **ml-modeling / technical-pattern**
- **Linear reductions bake into the decode matrices (≈64× smaller assets); nonlinear inverses cannot.** A zonal-mean or profile of an *identity-decoded* field is a linear operator → fold the lat/lon averaging into the matrices (≈29 KB vs 1.9 MB). A field needing a nonlinear inverse (exp/sigmoid) to reach physical units must decode the full grid, invert per cell, *then* average. Pick the asset only after you know which case you're in. — *`scripts/export_levelfields.py`* — **technical-pattern**
- **When parallel scouts contradict on a load-bearing fact, resolve it empirically — don't pick one.** Two scoping agents reported opposite vertical-level orientations (level 0 = surface vs level 0 = top-of-atmosphere). A one-line ground-truth probe — the warm + humid end is the surface — settled it and saved every level plot from rendering upside-down. — *`scripts/export_levelfields.py` (sigma-level orientation)* — **process / multi-agent**
- **Ship a cheap uncertainty proxy now — but label it for exactly what it is.** A per-cell σ across the five GCM lenses (model-choice *disagreement*) runs live client-side, captioned explicitly as "where the simulators disagree, **not** the emulator's own predictive error (that needs the probabilistic GPLFR model)." A proxy is honest only when it can't be mistaken for the posterior. — *`src/lib/emulator.ts` (`predictDisagreement`, Welford across lenses); `src/components/EmulatorPage.tsx`; commit `8b296df`* — **honesty**
- **Scope client-side feasibility by measuring the real artifact, not by assuming.** Whether the probabilistic emulator (GPLFR) could run in-browser was answerable only by reading its predict path and measuring its state: ~357 MB resident, an N×N kernel Cholesky, 64-sample posterior loops, and data-dependent control flow → not ONNX-able. The measurement both *ruled out* the obvious path and surfaced the cheap alternative. — *scope of `code/thousandworlds/models/_gplfr_core.py`, `gplfr.py`* — **process / scoping**
- **For a linear decoder, predictive σ factors — the expensive part collapses to one smooth scalar (shipped).** A GP's predictive variance depends only on the input, so per-cell σ becomes `σ[c] = √(base(x)·A_field[c])`, with `A_field[c] = Σ_j G_lat[c,j]²·Var(z_j)` (fixed per field) and `base(x) = 1 − k(x,X)K⁻¹k(X,x) ∈ [0,1]` — the GP density variance (0 at data → 1 far away). Only `base(x)` is world-dependent; it's distilled into a tiny 8→64→64→1 MLP (~230 KB JSON) so the client runs one forward pass — turning an intractable full-GP per-cell σ into a fully client-side layer. Verified in-browser: σ ≈ 17 K in-distribution → 23 K at the input-envelope corner (grows in extrapolation). Honest caveats kept in the UI: it's the PCA-GBT *family's* σ (not GPLFR's literal posterior), and the spatial pattern is fixed per field (only the magnitude responds). — *`scripts/export_uncertainty.py`; `public/uncertainty.json`; `src/lib/emulator.ts` (`predictUncertainty`); commit `b70ad01`* — **ml-modeling**
- **Survey (git log + grep) before re-entering shared code you stepped away from.** Six level-field plots had been built by a parallel session while away; checking the tree first meant extending the new plot-`kind` dispatch instead of clobbering it. — *`src/components/EmulatorPage.tsx` (plot-`kind`: map/profile/zonal/streamlines)* — **process**
- **Don't let a shared/absolute colour scale flatten low-variance data into a blob — default to per-item full contrast + mark the "special" item in CHROME, not in the data encoding.** Making a fixed absolute scale (160–380 K) the A/B-compare default squashed each temperate world's ~15 K of structure into a featureless orange sphere — a first-timer couldn't recognise it as the map they'd just seen, or tell the two worlds apart. Fix: render each world at its OWN full contrast (so it looks like itself, and the frozen world never shifts since its scale is its own), and mark the frozen/reference world with a *visual* cue (neon-emerald border + 🔒 chip) — never via a colour-scale trick. Keep absolute as an opt-in (with tooltip) + the difference map for the rigorous comparison. Generalises: when one of N compared items is "special" (frozen, selected, baseline), encode that in border/badge/icon; reserve the colour scale for the data, and never trade away per-item legibility for a shared axis a newcomer can't read. — *commit `4f09556` (per-world contrast + neon frozen marker); reversed the earlier absolute-default `521252f`* — **ux / data-viz**

It generalizes to any units pipeline, but stays a candidate until it clears the admission-policy bar.
