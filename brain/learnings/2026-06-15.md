# 2026-06-15 — Learnings

## The acknowledgments + credits-roll session — crediting as a discipline

**Setting**: built `ACKNOWLEDGMENTS.md` + an animated terminal credits roll (VHS), wired into the README, across many tight review cycles with the operator.

### What worked
- **Ship the asset *with its source*.** The credits GIF started as a throwaway `/tmp/credits.ts`, and the opening frame showed `bun /tmp/credits.ts` — a "this is a hack" tell. Fixing it the honest way (commit `scripts/credits.ts` + a `bun run credits` alias + the `.tape`) turned the GIF's opening command into a *real, runnable* one and made the whole roll reproducible. Lesson: a demo asset whose on-screen command doesn't actually work is a small principle-#13 violation; make the command real, and you get reproducibility for free (the ponytail "reproduce it yourself" ethos, applied to our own credits).
- **Proportionate credit is honesty in both directions.** Over-crediting a studied tool (calling a borrowed framework "the spine of it all") is as dishonest as under-crediting it. The fix: name each debt for exactly what it gave — generously and *specifically* — with no inflation in either direction.
- **Verify generative assets by frame, not by faith.** Every GIF regen was checked by extracting a frame with `ffmpeg` and *reading the image* before claiming it looked right — caught a line-wrap (too-narrow terminal) and the command-prompt tell that a "looks done" glance would have missed. Same discipline as the eval harness: look at the artifact, don't trust the generator.

### What surprised
- **Parallel-session drift is now the default, not the exception.** Origin moved on *nearly every push* this session — operator web edits (README License TL;DR, Builder Mode hero) plus parallel sessions (local-llm-setup product, /go-live, builder-mode command). One push faced a 4-local-ahead + 1-origin-ahead divergence; `fetch → rebase → never force` absorbed all of it clean, every time. The four pre-existing local commits (the operator's own unpushed work) rode up safely on the same rebase. **Rule reconfirmed: fetch + tip-check *immediately* before every push, treat origin as a moving target, never force.**
- **A changelog header can vanish in a rebase and read as "fine."** v1.26's entry body survived but its `##` header was dropped, silently re-filing a shipped feature (install.sh) under v1.27. By eye it looked complete. The durable fix wasn't renumbering (that would desync the changelog from commit messages + the v1.0.0 tag) — it was a one-time numbering note + a guard script (`bun run check-changelog`) that fails if any `vX.Y` in a commit subject lacks a changelog header. **Lesson: in a multi-session repo, treat the changelog as drift-prone state and guard it mechanically — a header is not proof the edit survived.**

### Status: green — ACKNOWLEDGMENTS + reproducible credits roll live; check-changelog guard live; all pushed, tree clean.

## A failed `git add` stages nothing — and "I pushed it" without verifying the diff is the real bug

**Setting**: renamed the hero GIF to bust GitHub's image cache, reported it fixed — it rendered blank. The README still pointed at the old (now-deleted) filename.

### Root cause (two compounding mistakes)
- **A multi-pathspec `git add` that includes ONE nonexistent path fails entirely and stages NOTHING from that invocation.** I ran `git add README.md hero.tape new.gif old.gif`, but `old.gif` had already been removed by `git mv`. The command errored on the missing path and silently left README + hero.tape **unstaged**. The commit still "succeeded" — it captured only the already-staged `git mv`. Net: the asset was renamed but its README reference wasn't → broken image, blank render.
- **Reporting "pushed/fixed" without verifying the commit's contents.** The deeper error: I claimed it was fixed without `git show --stat` / confirming the README diff actually landed. *A commit succeeding is not the intended change shipping.*

### Guardrails (every time, no exceptions)
1. **Verify the staged set against intent BEFORE committing** — `git diff --cached --name-only` (or `--stat`) must list every file you meant to change. A shorter-than-expected staged set = a failed/partial `git add`. Prefer staging files in separate `git add` calls, or `git add -A` for a known-clean tree, over one multi-pathspec call that can fail as a unit.
2. **Never report "done/pushed" without confirming the change is in the pushed commit** — `git show --stat HEAD`, or re-grep the file on `origin/main`. "It committed" ≠ "the change shipped."
3. **For asset renames, assert reference↔file consistency** — the file must exist AND every reference must resolve to it. This is mechanizable: a `check-assets` guard (sibling of `check-changelog`) that fails if any relative image/asset path in README/docs doesn't point to a real file would have caught this at commit time, before it ever shipped blank.

### Status: fixed (README ref + file consistent on origin). The class of bug is a candidate for a mechanical guard (recommended next).

## A scaffold template is only "done" when a fresh scaffold actually installs AND boots

**Setting**: launch-readiness audit before a Show HN. Ran the *real* new-user path — scaffold the starter → `bun install` → `bun dev` — instead of trusting it worked. (It hadn't been.)

### The incident
`bun install` on a fresh scaffold **failed at step one**: the starter's `package.json` listed `@inngest/sdk@^1.0.0` — a package that doesn't exist on npm (404) and was imported nowhere. *Every* product scaffolded from the template died on install; the headline "first win in 5 minutes" was dead on arrival to anyone who tried it. With it removed, the starter installs (741 pkgs) and `bun dev` boots in **3.8s** serving HTTP 200 with **zero env** — the claim was true, just blocked by one bogus line.

### Root cause
- The template **advertised an integration it never wired** (Inngest: a dep + env vars + a README mention, but no `lib/inngest.ts`, no usage) — and the dep name was wrong (`@inngest/sdk` vs the real `inngest`). Listing integrations you don't actually provide is both a principle-#13 honesty miss and the exact surface where this rot hides.
- It **was never runtime-verified.** The v1.25 changelog literally said *"the template can't be booted in-repo… the next scaffold is the live test"* — and that test was deferred, so the bug shipped. Reading the code ≠ running it.

### The rule (never again)
1. **A scaffold/template is NOT "done" until a fresh scaffold has been installed AND booted on a clean tree, end to end.** "The code reads fine" is not verification — same family as looks-done ≠ done. Before any launch that points strangers at the quickstart, run the quickstart yourself from zero.
2. **Only ship integrations the template actually wires.** Every optional integration is a dependency that can 404, mis-pin, or rot. No `lib/<x>.ts` using it → it doesn't belong in `package.json`, `.env.example`, or the README.
3. **Mechanize it.** The next guard (sibling of `check-changelog` / `check-assets`): a `check-starter` that scaffolds + `bun install`s (+ optionally boots) the template and fails on any error — run in CI so a broken/nonexistent dep fails before it reaches a user. Recommended as part of the CI workflow (#6).

### Status: template fixed + pushed (Inngest removed entirely; install + boot verified). No scaffolded products affected (code-paths empty; every other hit was a template copy). Mechanical guard recommended next.
