# 2026-06-29 — Learnings

## A site that looks "fully static / no backend" can have an ENV-GATED backend — never conclude it from code alone

**Setting**: A site/scale audit of **Thousand Worlds Explorer**. The first pass read the code in a **keyless / dev state** and concluded: *fully static, no backend, no DB, "scales to 10k trivially."* That was **wrong**. The site actually has a **live Supabase backend** — magic-link + Google OAuth, a `creations` CRUD table with share slugs, a `profiles` / `is_admin` table, an `events` telemetry table, and an AdminDashboard — **plus** a `/emulator` route that proxy-rewrites to a *separate* Vercel app. The Supabase client only **initializes when env vars are present** and degrades to local-only when they're absent, so the keyless first pass saw none of it. The keyed/production state is the truth; the audit graded the wrong state.

### The lesson (the rule)

A site can look fully STATIC / "no backend" when its backend is **env-gated**. Audit it in a keyless/dev state and you'll wrongly conclude it has no backend and "scales trivially." **Never conclude "fully static / no backend" from code alone — verify the env/keys and the production deployment state.** Conditional code (`if (SUPABASE_URL) initClient()`) is a *live backend* in prod, not a dead branch.

### Why it bit (what the first pass missed)

- The backend SDK was **initialized conditionally**, so a keyless read showed graceful local-only fallback and *looked* like "no network, no DB."
- "Static" is a **delivery claim, not a code-only property** — the same bundle is static in dev and backend-driven in prod purely because of which env vars are set.
- A "static" route (`/emulator`) was actually a **proxy rewrite** fronting a separate deployment with its own scaling limits — invisible unless you read the hosting config.
- "Scales to 10k trivially" skipped every real scale ceiling: **free magic-link SMTP rate limits** (the classic first failure), **DB tier caps** (connections / auto-pause / storage), and **RLS / anon-key authorization** correctness.

### Mandatory checks this injects into any site / scale / production-readiness audit (REQUIRED, not optional)

1. **Read the env files and trace the gating.** `.env`, `.env.local`, `.env.production`, `.env.example` — populated keys mean the backend is **LIVE in prod**, even when the code path is conditional.
2. **Grep deps + src for backend SDKs even when usage is conditional**: `supabase`, `@clerk`, `firebase`, auth libraries, `prisma`/`drizzle`/any DB client, server SDKs.
3. **Check hosting config** (`vercel.json`, `netlify.toml`, `next.config.*`) for **rewrites/proxies to OTHER deployments** — a "static route" can front a separate app with its own scaling limits.
4. **Distinguish BUILD-TIME vs per-user RUNTIME** network calls. A call at build/SSG time is not the same risk as one on every user request.
5. **If a backend exists, audit the scale-critical layers**: **auth email/SMTP rate limits** (free magic-link SMTP is the classic FIRST failure at scale), **DB tier limits** (connection caps, auto-pause, storage ceilings), and **RLS / authorization correctness** (public anon keys get probed at scale).

### First-timer heuristic (verbatim — keep it simple, this is the door for new builders)

> **If the site has a sign-in button, a save/share feature, or an admin page, it HAS a backend — even if the code looks like it runs without one. Open the .env file and check.**

### Where it's enforced now

- **Anti-pattern**: [[concluding-no-backend-from-code-alone]] — the canonical "don't even consider it" wound, surfaced in `BEST-PRACTICES.md` under *Never do this*.
- **Required audit step**: `factory/playbooks/mvp-stage/security-checklist.md` opens with a **"Backend reality check"** section (the 5 checks + the first-timer heuristic) — it runs *before* the auth/RLS sections, which assume a backend already exists.
- **Auto-enforced gate**: `/security-check` (`factory/commands/security-check.md`) runs the backend-reality check first and can BLOCK on a "claimed static but env-gated backend" finding.
- **Site-launch library**: a matching gotcha in the `launch-gotchas` skill so the next launch learns it without being burned.
