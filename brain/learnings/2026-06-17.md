# 2026-06-17 — Learnings

Session: Patently Legal — scaling `case_law_index` from 1,586 → 25,000+ to sit credibly beside 50,000+ patents. The cursor-pagination fix ([2026-06-16](2026-06-16.md)) unfroze the corpus, but reaching scale surfaced two harder walls and one clean pivot. All proven on a live product.

## A public legal API's rate limit can make API-based corpus seeding impossible — use the BULK DATA dump instead

**Setting**: with cursor pagination fixed, the CourtListener *search* API finally returned distinct pages — but the count crawled, then stalled completely.

### Root cause
CourtListener throttles the API to **125 requests/day** on our token (confirmed via the 429 body: `"Rate limit exceeded: 125/day. Expected available in 52162 seconds"`). The limit is **global to the token across every endpoint** (search, clusters, opinions, dockets — all 429 once exhausted). At 20 results/page that's a hard ceiling of ~2,500 cases/day → 25k would take ~10 days. The grind wasn't slow, it was *capped*.

### The pivot — bulk data, not the API
CourtListener (like many public-data orgs) publishes **quarterly bulk CSV dumps on S3** with **no rate limit**. That's the intended path for what we were doing. The build:
- **`nature_of_suit` lives on the DOCKET, not the opinion-cluster** — the clusters dump had 27 IP hits in 11M rows (essentially unpopulated). So a naive "filter clusters by NOS" yields nothing. The fix is a **two-stream join**: stream the dockets dump (~63M rows, 4.6GB) keeping `court_id=cafc` (all patent/IP appeals) + IP `nature_of_suit` docket_ids → then stream the clusters dump keeping rows whose `docket_id` is in that set. Surfaced ~42k IP cases → embedded the best ~26k (precedential + most-cited first).
- Result: **1,586 → 28,000+ in one overnight**, vs ~10 days via the API.

### Guardrails
1. **Before building an API-paginated ingest for *bulk* corpus, check for a bulk-data dump.** APIs are for interactive/incremental use; their rate limits are sized for that. A bulk dump (S3/torrent/Postgres COPY) sidesteps the limit entirely and is the maintainers' intended path. Always read the provider's "bulk data" / "data downloads" help page first.
2. **Probe the real rate-limit ceiling early** — one request, read the `429` body + `Retry-After`/`X-RateLimit-*` headers. "125/day" vs "5000/hour" completely changes the architecture; don't discover it 1,000 wasted requests in.
3. **Know where your filter field actually lives.** A denormalized field (here `nature_of_suit` on clusters) can be present-but-empty; verify population on a sample before designing the whole filter around it. The authoritative copy was on dockets.

## Agent-spawned background jobs get reaped — durable long-running work belongs in the user's own terminal (or must be resumable + watched)

**Setting**: ran the multi-hour bulk extract/load as `nohup … & disown` from the agent's sandbox shell.

### What bit us
`nohup` survives SIGHUP but **not** the sandbox/session lifecycle — agent-spawned processes can be reaped when the turn/session ends, and they're invisible to the harness (not task-tracked), so they need manual polling. A multi-hour job started this way can silently die.

### Guardrails
1. **For durable long-running jobs (corpus seeds, migrations), prefer the user's own terminal** — it outlives the agent session. If it must run agent-side, make it **fully idempotent + resumable** (skip-existing, checkpoint), poll it on a schedule, and relaunch on death.
2. **Idempotency is non-negotiable for any seed.** `onConflictDoNothing` + skip-already-present made every restart safe and cheap — we relaunched the loader ~5× across pauses/deaths with zero double-work.
3. **Guard inserts against a free-tier DB ceiling.** The loader checked `pg_database_size` every N rows and stopped at a 490MB cap (512MB free tier) — silent insert-drop-when-full is a known Neon failure mode; a live size check turns it into a clean stop.

## Live app and a background embedder fight over the same vendor + DB

**Setting**: while the bulk loader hammered Voyage embeddings + Neon writes, the live chat/clearance pipelines (which also embed the query + read vectors) slowed to a crawl and **timed out past their serverless `maxDuration`** — clearance got killed before marking itself complete (stuck "running" forever), chat's stream was cut off mid-answer.

### Guardrails
1. **A heavy background job and the live request path share vendor rate limits and DB compute even in separate processes.** The in-memory throttle is per-process, but Voyage's account TPM and Neon's compute are shared — so the batch job starves live traffic. Pause batch work during active use, or rate-limit it to leave headroom.
2. **Long pipelines in a serverless request are fragile — a timeout orphans state.** Raise `maxDuration` to cover the realistic worst case (chat 60s→300s here), and/or persist progress incrementally so a kill doesn't lose the run. A run that can't finish reads as "broken," not "slow."
