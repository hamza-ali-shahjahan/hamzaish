# Learnings — 2026-07-04

## Graded the factory on Osmani's agentic-autonomy scale — the two-axis split is the useful part

- Ran Hamzaish against Addy Osmani's 6-level autonomy model (agency axis vs orchestration axis, https://addyo.substack.com/p/agentic-autonomy-levels). Verdict distilled in `brain/knowledge/2026-07-04-autonomy-grade-osmani-scale.md`.
- **What worked:** the factory's existing command ladder maps cleanly onto the agency axis with no retrofitting — `/full-cycle` = L1 gates, `/auto` = L2 scoped delegation, `/goal` = L3 goal-driven (rubric + fresh-eyes verifier is exactly Osmani's "contract": goal, scope, stopping condition, evidence, budget). Risk-calibrated gates (irreversible/outward-facing actions always human) match the article's "risk and reversibility set the ceiling" rule — we arrived at the same posture independently.
- **What surprised:** the honest orchestration grade is lower than the agent-count suggests. ~30 stage agents exist but one session routes them sequentially; `meta/parallel-sessions-protocol.md` deliberately caps at one session per product; `scripts/autonomy-loop.ts` relaunches *serially*. That's calibration, not weakness — but it means L4 "parallel delegation" is scaffolding, not practice. Applying our own Hard Rule #1 discipline (never claim PMF from launch-week numbers) to autonomy: don't claim a level whose metric we haven't sustained.
- **Gap named for the next rung (one axis at a time, orchestration first):** a trigger-driven manager — cron/portfolio-pulse dispatching `/goal` runs and escalating only on `blocked` — is the L4→L5 path. Agency is already at its risk-appropriate ceiling; climbing it further would be autonomy-as-status.

## Deep-researched the jdx/en.dev toolchain (mise, fnox, hk, pitchfork, aube) for factory fit — verdict: a ~1.5x infra layer, not a 100x

- 106-agent verified research pass (23 confirmed claims, 2 refuted). Full analysis delivered in chat; distilled here.
- **Highest-value finding:** fnox's MCP server (`fnox mcp`, exec-only allowlist) is a *root-cause* fix for the 2026-07-03 secrets-leak class — secrets never exist as plaintext on disk (ciphertext or provider references in committable `fnox.toml`), and the MCP `exec` tool redacts resolved values from stdout/stderr (`printenv` → `[REDACTED]`). Our guard hook patches the symptom; fnox removes the plaintext file entirely. Caveat: redaction is literal string-matching (base64 transforms can leak) and defaults are permissive until `tools=["exec"]` + secrets allowlists are set.
- **Second:** pitchfork v2.10–2.15 (May–Jul 2026) added per-worktree HTTPS subdomains, auto-trusted CA, daemon groups, and its own MCP server — the right shape for N parallel product dev servers vs our ad-hoc `preview_start`. Young (~550 stars) but 37signals/entire.io-sponsored.
- **Surprise:** aube is a Rust Node package manager ("never forget to install"), not an AI tool — mostly irrelevant to our Bun-first stack. hk solves git-hook parallelism; our guards are *Claude Code* hooks, a different layer, so hk only helps product repos.
- **The honest multiplier discipline held:** tools compress the loop that is already fast (build); the binding constraints are distribution, decision bandwidth, and autonomy grade — same conclusion as the Osmani grading above, arrived at from the opposite direction.

## Reviewing the fnox wiring caught two real defects — verify config-file behavior, don't assume it

- Self-review of the Phase 1 fnox wiring (prompted by the operator) caught two shipped bugs before commit:
  1. **`_comment` keys break strict config files.** I'd put `"_comment": "..."` in the starter's committed `.claude/settings.json` and `.mcp.json.example`. Per Claude Code docs, project/local settings use *strict* validation — an unknown key "is rejected as a whole." The deny-rules would have **silently not loaded**. Fix: strip `_comment` from all loaded/copied JSON; document rationale in SETUP.md + the skill instead. **Lesson: JSON config for the harness gets no inline comments; the doc lives elsewhere.**
  2. **Deny-rules are best-effort, not a boundary — and arg-constrained ones are the fragile kind.** My first cut denied `Bash(fnox get:*)`, `Bash(fnox export:*)`, etc. The docs explicitly warn arg-constrained Bash patterns are bypassable (option injection `fnox --version get`, env prefixes, `eval`, absolute paths). Fix: block the *whole tool* (`Bash(fnox)` + `Bash(fnox:*)`) since the agent has zero legitimate shell use of fnox, AND reframe every doc so the *real* boundary is "decryption key out of the agent's reach" (remote-provider auth / unreadable key file), with deny-rules demoted to best-effort defense-in-depth.
- **Meta-lesson (reinforces [[verify-provenance-before-claiming]]):** I *behaviorally* verified the tool (red-teamed the MCP) but *assumed* the config-file plumbing. The plumbing had the bugs. Verify the boring integration layer too — a correct tool wired through a silently-rejected settings file is an un-enforced control.
- Also fixed coherence gaps: the go-live guided loop (steps 3–5) and SETUP.md were still `.env.local`-only after fnox became the default; made them backend-aware. Verified `fnox set KEY` (no value) prompts with hidden input and `fnox list`/`check` never print plaintext — so the "value never in chat / non-printing verify" claims are accurate, not aspirational.

## Phase 2 shipped: pitchfork as the supervised dev-server floor — and the schema-validation lesson repeated

- Piloted pitchfork in scratchpad (throwaway daemons — `code-paths.local.json` is empty so all products are showcase; no real repos touched). Verified: start-once idempotency, HTTP ready-check, cross-session persistence via the supervisor, crash detection (errored + logs), and an MCP server exposing `pitchfork_start/stop/restart/status/logs` for agents. Wired an opt-in `pitchfork.toml.example` + MCP entry into the starter, and made `/go-live` verify a server is up before sharing a localhost link.
- **The Phase 1 lesson repeated, and I caught it the same way:** my first `pitchfork.toml` guessed `ready = { http = ... }` — pitchfork **silently accepted and ignored it** (lenient parser). Checked the README: the real fields are `run`/`auto`/`ready_delay`/`ready_output`/`depends`/`cron`. Same failure shape as fnox's `_comment` (strict) and `identity_file` (wrong name). **Standing rule now: never ship a tool's config file without validating field names against its primary source — "it parsed" proves nothing.**
- **New honest limit found by red-teaming, not docs:** pitchfork's HTTP ready-check **false-positives to "running"** when another process already answers the port — a 2nd daemon on a bound port reported healthy while its process had died (`Errno 48` in its own logs). **Readiness ≠ liveness.** Mitigation baked into the template: `ready_output` (matches the daemon's own stdout) + distinct ports per parallel product.
- **Escalation discipline held:** did NOT run `pitchfork proxy` (it installs a system CA + edits `/etc/hosts`) — documented it as operator-gated instead of executing it. Same posture as never touching secrets files.
- Both phases (fnox 2.6.0 + pitchfork 2.7.0) staged, uncommitted, batched for one PR pending an explicit ship go-ahead. The jdx-toolchain floor is the two halves of "safe unattended runs" — the prerequisite the autonomy grading named for the orchestration rung.

## Phase 3 "go" → the manager loop already existed; the real work was wiring the floor into it (and self-enforcing the contract)

- "Let's go with Phase 3" → before building a manager loop, I read the tree: `scripts/autonomy-loop.ts` ALREADY is the unattended `/goal` manager (opt-in `.autonomy-ok`, STOP switch, session budget, `achieved/blocked/continue` handoff, `--dry-run`), and `factory/model-policy.ts` ALREADY does #4 model routing (spawn-boundary, stakes-escalation to Opus). **Building a new harness would have been pure duplication.** The `references discipline` / "verify what exists before building" paid off yet again.
- So Phase 3's real increment was small and surgical: wire the v2.7.0 floor into the existing loop — a fnox-config gate + pitchfork readiness note + a loud spend-visibility (#6) warning — and upgrade the loop's PASSIVE escalation (a console line nobody watches unattended) to ACTIVE (durable `ESCALATION.md` + desktop notification). The gate makes the contract's hard preconditions **self-enforcing** — policy→enforcement, same pattern as the secrets hook.
- **The honest wall held:** the live unattended pilot genuinely can't run yet — `code-paths.local.json` is empty (no registered product) and #6 spend visibility isn't built (efficiency session's lane). That's not stalling; the gate I built now *enforces* that wall. Don't fake a pilot on a product that doesn't exist.
- **Self-caught bugs (the session's recurring class, third time):** my first gate `✓ fnox: secrets resolve` — but `fnox check` verifies secrets are DEFINED, not that they DECRYPT (removing the key still passed). And `✓ pitchfork: reachable` — but `pitchfork status` exits 0 for a merely DEFINED daemon (readiness≠liveness again). Testing exposed both; relabeled to say only what's verified. **Lesson restated: a check that "passes" is worthless until you've proven it fails when it should.**
