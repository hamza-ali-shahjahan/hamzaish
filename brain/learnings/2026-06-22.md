# 2026-06-22 — Learnings

## Parallel agent sessions: the shared *working tree* is the collision surface — isolate with git worktrees

**Setting**: multiple Claude sessions working the same product (`local-llm-setup`) concurrently.
One session shipped a UI uplift (light/dark theme); another was mid-flight on a Visual-RAG
("Knowledge") feature. The operator asked to make **branch-first** a permanent, Hamzaish-level
cadence — captured as Operating Principle #14.

### What we learned
- **Direct-to-main is not the only clash risk — the shared *working tree* is worse.** Both sessions
  edited the *same* files: the repo's `local-llm-setup.sh/.ps1` **and** the shared dogfood source
  outside git (`~/.local-llm-setup/chat/index.html`, `agent-server.py`). The other session's
  uncommitted WIP (291 installer lines + 55 agent-server lines + new `tools/visual_rag.py`) was just
  *sitting* in the tree. A naive "bake everything + commit" would have swept all of it into the UI
  release. **Git branches alone don't help if both branches check out the same working directory.**
- **The fix is a git worktree per session.** `git worktree add ../sibling -b <branch> origin/main`
  gives a clean, independent checkout off latest main — it carries **none** of the other tree's
  uncommitted changes. Build, bake, commit, push, PR there; the other session's folder is never
  touched. Merge via the GitHub API (`gh pr merge`) so the shared local tree isn't disturbed; don't
  `git pull` into a tree that holds someone else's WIP.
- **Region-scoped bake when the source is shared + contaminated.** The dogfood `index.html` ended up
  a *mix* of my theme work and their Knowledge/embedding UI. Solution: don't bake the dogfood
  wholesale — extract the **clean** embedded HTML from `origin/main`'s installer, apply only my
  change to it, and splice **just my region** (the builder HTML block, not the agent-server block)
  into the worktree. Then assert the result contains **0** of the other feature's markers before
  committing. Byte-verify the spliced block.
- **A half-built parallel feature leaks into the live product.** Because the dogfood file is served
  live, the other session's unfinished "Knowledge" button showed up in the operator's builder and
  was *context-unaware* (its own vision-check disagreed with the shipped Capabilities `/api/tags`
  detection). Unmerged WIP on a shared live file = the user sees half-finished features. Another
  reason to commit WIP to its own branch and isolate trees.

### What shipped
- **Operating Principle #14 (branch-first + per-session worktrees)** added to the brain.
- local-llm-setup **v1.19.0** (light/dark theme) and **v1.19.1** (Capabilities: smaller tiers read
  "Optional" when a bigger one is installed) — both built in isolated worktrees off `origin/main`,
  region-scoped bakes, verified **0** parallel-session WIP leaked, CI-green, paused at PR for the
  operator's go.

### Surprise (highest signal): the cure was an OS/git primitive, not discipline. "Be careful not to
commit the other session's changes" is unenforceable across concurrent agents. `git worktree` makes
the clash **physically impossible** — each session gets its own directory, so there's nothing to be
careful about. Reach for the structural fix (separate trees) over the behavioural one (remember not
to clobber).
