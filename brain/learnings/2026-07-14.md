# Learnings — 2026-07-14

## The factory's biggest tax was invisible until measured: ~3.8k tokens of skill descriptions per session

Studying mattpocock/skills surfaced a frame Hamzaish lacked: every model-invoked skill's description is context load paid *every turn of every session*, and the alternative currency is cognitive load (user-invoked skills the operator must remember — which our CLAUDE.md command table already amortizes). First audit measured 42 skill + 24 command descriptions at ~3.8k tokens/session, and found ~10 names listed **twice** in live sessions. The meta-lesson: a factory that writes skills without a skill-writing standard accumulates sediment silently — the standard is now `factory/playbooks/ai-native-2026/skill-authoring.md`. Rules worth keeping: **(1) invocation mode is the first authoring decision, not an afterthought; (2) measure context load before and after any skill edit — it's the factory's rent.**

## A diagnosis that pattern-matches an anti-pattern still needs verification: `ls -la` follows directory symlinks silently

The double-listing was first blamed on `.claude/` being hand-copied files drifted from `factory/` — it fit the hand-maintained-facts-drift anti-pattern perfectly, and `ls -la .claude/commands/` "confirmed" regular files. Wrong: `.claude/commands` and `.claude/skills` are intact **directory-level symlinks**, and `ls` on a dir symlink transparently lists the target's contents as plain files. `readlink` (or `git check-ignore`'s "beyond a symbolic link" error, or `git ls-files`) told the truth in one command. The real, verified cause of at least some duplicates: **skill/command name collisions** (`factory/skills/brain-ask/` + `factory/commands/brain-ask.md` both exist). Rules: **(1) verify symlink wiring with `readlink`/`git ls-files`, never `ls`; (2) when a finding perfectly matches a known anti-pattern, that's a reason to verify harder, not less — pattern-match is a hypothesis, not evidence.**

## Porting a meta-skill is higher leverage than porting any object-skill

Of 18 candidate skills in mattpocock/skills, the one adopted first was the one about writing skills — because Hamzaish's output *is* skills, so the meta-layer multiplies everything downstream (same logic as `meta/factory-improving-factory.md`). The object-level skills (wayfinder's fog-of-war decision maps, CONTEXT.md ubiquitous language, triage's `.out-of-scope/` rejected-concepts KB) were deliberately left as separately-decidable adoptions rather than bundled in — one decision per adoption keeps each reversible and reviewable. Leading words worth stealing even before their skills are ported: *fog of war* (chart only what you can state precisely), *decision ticket* (resolves a decision, not a build slice), *red-capable* (a loop that can catch this specific bug), *tight* (fast + deterministic + sharp).

## Ship-time security gates decay at scale — coverage must be continuous, and data minimal

An outside prompt (the 7TB/11B-record open-server disclosure) stress-tested the factory's security layer and found it strong exactly where it fires (ship time: `/security-check`, security-reviewer, the secure-by-default scaffold) and blind everywhere it doesn't: nothing watched the *running* system for drift (a dashboard-added table without RLS, a bucket flipped public never touches the repo, so repo-greps can't see it), the incident playbook covered outages but not breaches (different endgame: users safe and informed, not service resumed), and nothing bounded what data accumulates (11B records is hoarding before it's an auth failure). Closed with `factory/playbooks/scale-stage/security-at-scale.md` + `/security-check --live` + a retention rule in the spec template. The general rule: **any gate that fires on an event (deploy, PR, scaffold) verifies a snapshot; anything that mutates outside that event needs a scheduled re-verify.** Audit every other gate in the factory against this shape.
