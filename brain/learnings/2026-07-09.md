# Learnings — 2026-07-09

## A stalled repo with a task plan revives in a day; one without would have died

dnsdoctor sat untouched since May at 4/20 tasks — and went to code-complete (185 tests, 48 e2e, 248-page production build) in ~24 hours across five chunked build sessions. The revival cost almost nothing because the repo carried its own resurrection kit: a SPEC, a dependency-graphed task plan with per-task acceptance criteria, and a commit style that made progress legible from `git log` alone. The one desync: `tasks/todo.md` checkboxes were never ticked, so the *file* said 0/20 while git said 4/20 — git log was the truth. Rules worth keeping: **(1) a task plan with acceptance criteria is a product's hibernation insurance; (2) tick the todo in the same commit as the task, or the plan file rots into a liar; (3) on revival, trust commits over checklists.**

## Chunked sequential build agents work when the handoff is explicit

Five background agents built tasks 5–20 with zero conflicts and zero rework. What made it work: 3-task chunks (agent context stays sharp), each agent's report ends with "notes for the next tasks," and the orchestrator pastes those notes verbatim into the next agent's brief (the '@'-apex convention, the `vi.hoisted` mock pattern, the `/^cop(y|ied)$/i` locator lesson traveled agent-to-agent this way). Also load-bearing: one commit per task in a fixed style, "impossible-as-written criteria → closest honest version + declared deviation, keep going" (prevented every stall), and an explicit skip-contract for key-gated live tests ("exactly 4 skips" — any drift is a signal, not noise). Candidate for `factory/workflows/` if this pattern runs a third time.

## Agents surface honesty bugs specs miss — when told deviations are acceptable

Three examples from one build: the plan's Google Workspace "multi-MX" quirk was the pre-2023 legacy scheme (agent encoded the real single-MX gotcha and said so); a streaming + `loading.tsx` interaction made invalid domains soft-404 with HTTP 200 (agent moved validation to the layout for a genuine 404); the "healthy domain → ok" live assertion was flagged as model-behavior-dependent before it ever ran. The common enabler: the brief *licensed* declared deviations. An agent ordered to satisfy acceptance criteria literally would have shipped the outdated record scheme and the fake 404s. Rule: **in build briefs, always pair "meet the criteria" with "the closest honest version, deviation declared, beats literal compliance."**

## The demo-server-alive pattern: hot-reload makes "keep building" and "let the operator click around" compatible

Running the operator's demo on `next dev` while agents kept committing worked because hot-reload turned every landed task into a silently-updated demo, and `reuseExistingServer: true` let Playwright ride the same server. The one prerequisite: tell each agent the server exists, that it must be serving 200 at the end, and exactly how to restart it if a build wipes `.next`. Verified across four consecutive chunks without the operator ever losing the link.
