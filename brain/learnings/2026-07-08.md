# Learnings — 2026-07-08

## Live Path M1 shipped — the gap between "commands returned 0" and "provably live" is now a script

- Built `scripts/verify-live.ts` (the A1–A10 harness) and wired it as `/go-live`'s blocking close condition. Shipped as v2.11.0.
- **What worked:** smoking the harness against a *real* live product (ventbox.co) before shipping caught a verdict-design bug the mocks never would have: a Next.js app with no health route answers `/api/health` with a **200 catch-all page**, and the first draft called that FAIL. The honest verdict is PENDING — "this product predates the convention" — because FAIL must be reserved for a real health endpoint reporting sick. Rule worth keeping: **an assertion harness needs one run against a messy real target before its verdicts can be trusted; mocks only confirm the failure modes you already imagined.**
- **What surprised:** the five-verdict split (PASS/FAIL/PENDING/MANUAL/SKIP) did more design work than the assertions themselves. The playbook's binary pass/fail couldn't express "no failures, but you haven't created the throwaway user yet" — and that in-between state is exactly where "declared live" lies happen. MANUAL items printed with remediation keep the human steps honest instead of silently dropped.
- **Convention gap found and closed:** the starter template had no `/api/health` at all, so A3/A4/A6 were unassertable on every product the factory has ever scaffolded. The playbook had assumed a convention nobody had built. When a playbook references an endpoint/file/flag, check it exists in the template that's supposed to provide it — specs drift ahead of scaffolds.
- **Parallel-session note:** mid-arc, another session released v2.10.0 (tag + changelog entry) and committed files this session had authored earlier (the 07-07 learning). The curated-staging discipline from the /pr flow absorbed it with zero conflict — branch from `origin/main`, stage an explicit file list, re-read `git status` before every ship. That discipline is why two sessions can work the same repo without stepping on each other.

## Studied stablyai/orca (12.1k stars in ~110 days, zero launches) — distilled into the factory

- **Verify growth claims against the API before believing them.** The founder's "10k stars, no launch" account checked out exactly: repo created 2026-03-17, ~12.1k stars, ~5,080 merged PRs (~46/day, agent-driven), 859/1,255 issues closed, several releases per day, community PRs auto-tracked by a bot on open. The mechanics are infrastructure, not habits — scheduled release cuts with idempotent slot markers, version safety gates, a ~100-line community-PR Action.
- **What surprised:** their `skills/orchestration/SKILL.md` spends most of its words on *when NOT to use it* — handoff vs. supervision, stated as protocol. The inverse of how most skills are written (all capability, no routing), and it's why their agents route right. Promoted: two playbooks (`community-flywheel.md`, `handoff-vs-supervision.md`), a routing-hygiene meta-rule, negative-routing lines on the four collision-prone commands, /release heartbeat + safety gates, and the naming/why-comment legibility rule in AGENTS.md + starter template.
- **Dedup check paid off:** the "ship visibly" lever already existed as `release-cadence-as-content.md` (immich/AppFlowy research) — amended it with the Orca proof point instead of writing a duplicate. Before adding a playbook, grep the stage folder for the lever it teaches.
- **Kept out deliberately:** Orca's runtime (Electron shell, terminal persistence, worktree UI, mobile RPC) — Claude Code already provides that layer, and Hamzaish is an agent OS, not a runtime. Mission check held: every adoption is a playbook or rail a builder can reuse, not machinery that changes what Hamzaish is.

## Prospect-research sweep (customer discovery) — channel yields, measured

- Ran a 3-agent parallel sweep for "non-technical AI-builder with post-build distribution stall" prospects. Yields: **Reddit builder subs 12/12 verified** (r/vibecoding is the richest single sub; Reddit's own search JSON + the Arctic Shift archive API work where Google/DDG exact-phrase queries return nothing), **indie communities 12** (Indie Hackers posts fetch directly), **X: ~0 verifiable** — logged-out X only surfaces big accounts; the exact target persona (small account, 8-view tweets) is invisible to web search. Rule for the cold-outreach agent: **for persona mining, Reddit + IH first; X only logged-in with saved queries** (reply-mining a tool vendor's "what did you build?" thread beats keyword search).
- Verification discipline paid: both agents dropped tempting title-only/image-only posts ("Zero users. Zero revenue.") as unverifiable rather than padding the list — the deliverable states "fewer-but-real beats padded" explicitly in the prompt, and that line did the work.
