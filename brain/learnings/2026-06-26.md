# 2026-06-26 — Learnings

## Always share the live + localhost links at the end of every response

**Setting**: ThousandWorlds Explorer session — a Vite/React app deployed on Vercel, public GitHub repo, local dev server on :5173. The user escalated (again) after I finished substantive turns without surfacing the URLs they'd want to click (the localhost link AND the live Vercel link). They made it a universal standing rule across all conversations and projects, and asked to build it into Hamzaish.

### The rule (now global)

Whenever a conversation or build involves a **localhost** OR **any deployed/live link** (Vercel, Netlify, GitHub Pages, a custom domain, a public repo, a dashboard — any URL where the thing lives), **end the response with those links**, clearly labeled, on their own line(s). Share BOTH the live link AND the localhost link when both exist — never just one. Hard default for every main conversational turn where such a link exists, not only when asked.

### Why it kept happening

The prior global rule was scoped narrowly to "localhost for local builds," so deployed-only and repo links slipped, and on discussion-heavy turns the links felt optional. Reframe that fixed it: surfacing the links is a **required finishing step** of the response — like a build that isn't done until it's verified — not a courtesy.

### Where it's enforced now

- Global `~/.claude/CLAUDE.md` — broadened the "share the links" rule to cover all links at the end of every response (all sessions, all projects).
- This ledger — `BEST-PRACTICES.md` → *Run the factory* — so factory/Hamzaish builds carry it.
- No clean per-tool trigger exists for a hook (unlike the AskUserQuestion recommendation guard, which was hookable), so this one is instruction-enforced; offered the user an optional `Stop`-hook nudge as belt-and-suspenders.

### Decision

Standing communication default for all Hamzaish-driven work: every handoff/response ends with the clickable links (live + local + repo as relevant).

## Verify a localhost link actually serves before sharing it (the sandbox preview ≠ the user's localhost)

**Setting**: Same session. I'd been pasting `http://localhost:5173` at the end of turns, but the user reported it was dead. Diagnosis (`curl` + `lsof`): nothing was listening on 5173 — the preview tool I use to verify UI changes runs the dev server in a **sandbox the user's own browser can't reach**. So the link was never live for them; I'd been handing over a dead link while believing it was up.

**Lesson**: "make sure the server is running" means a REAL dev server on the user's machine, **verified** — not the harness's sandboxed preview. Before sharing a localhost link: start the actual dev server (`npm run dev` as a background process) and confirm the port returns HTTP 200 (`curl -sS -o /dev/null -w "%{http_code}" http://localhost:PORT/`). Only paste the link once it answers 200.

**Bonus**: when the user asks to run it themselves, teach the manual terminal flow as baby steps — open Terminal → `cd` to the folder → `npm run dev` → open the printed `Local:` URL → `Ctrl+C` to stop — with the port-in-use (Vite jumps to 5174), `npm: command not found` (install Node), and blank-page (hard-refresh) troubleshooting.

**Decision**: folded into the global "share the links" rule (the localhost bullet now says *verify it serves from a real server, HTTP 200; the sandbox doesn't count*) and the BEST-PRACTICES ledger bullet.

## A "my changes reverted!" scare that was really an HMR state-reset from a parallel session

**Setting**: Building a big *uncommitted* UI rewrite (the Imagine Lab) while a **parallel Claude session was editing the SAME working directory**. The parallel session committed a change to a sibling file (`ThousandWorlds.tsx`) that my file imports a type from.

**Symptom → wrong conclusion**: Right after, a verification eval suddenly returned all-false (the new UI elements "gone"), and a harness "file modified by a linter" note displayed the OLD version of my file. I concluded my uncommitted redesign had been reverted and nearly re-wrote it from scratch.

**Reality**: Nothing was reverted. `git status` + a `grep` for new-vs-old markers showed my new code was intact on disk the whole time. The parallel commit to the imported sibling triggered Vite **HMR, which remounted my component and reset its React state** (selected planet → empty), so the result card vanished; my multi-step eval ran *during the reload churn* and clicked a stale/rebuilding DOM → all-false. The "old content" in the system note was a stale/misleading snapshot, not the disk truth.

**The fix that worked**: (1) Checked the actual disk/git state (`git status`, `grep` the file) instead of trusting the alarming signals — confirmed the code was fine. (2) **Committed the work-in-progress immediately** to protect it from the active parallel session, then pushed. (3) Hard-reloaded the preview and re-picked → everything worked.

**Lessons to bake in**:
- **When a verification suddenly shows "everything broke," check `git status` + grep the file on disk BEFORE concluding a revert.** The cause is almost always an HMR state-reset or a misleading tool note — not data loss. Don't panic-rewrite.
- **HMR remounts components and RESETS React state on any dependency change** (not just stale canvas draw closures). A screenshot/eval taken right after an edit — or after a *parallel* edit to an imported file — can show empty/stale state. Hard-reload and re-establish state before trusting it.
- **A parallel session on the same working dir is a real hazard**: its commits land on your branch, its edits trigger your HMR, and it can race your uncommitted work. Detect it (unexpected reflog commits, "file modified" notes) and **commit WIP promptly + push** to claim your work.

**Decision**: folded into the BEST-PRACTICES "Run the factory" ledger; verify-on-disk-before-panicking and commit-WIP-when-sharing-a-dir are now standard reflexes.

## Claude Code 2.1.x merged commands→skills, and its loader skips symlinks — de-symlink your commands

**Setting**: After a Claude Code update (2.1.191), `/hamzaish` returned "Unknown command" — yet the command file was healthy (real file, valid frontmatter, clean encoding). A parallel session was mid-fix ("registering it as a skill"); took over to avoid a race on the same files.

**Root cause**: 2.1.x folded slash-commands and skills into ONE loader (skills-first; global `~/.claude/agent-skills/skills/`, project `.claude/skills/`, legacy `.claude/commands/` for back-compat). Two traps at once: (1) the maintainer's global `~/.claude/commands/*.md` were **symlinks into the repo**, and the new loader doesn't reliably *follow symlinks* during discovery; (2) an in-project session sees the commands (real file on disk) while a session launched elsewhere relies on those global symlinks → "Unknown command."

**Fix**: de-symlinked — replaced the global `~/.claude/commands/*.md` symlinks with **real files** (`cp -L`), so no symlink-following is needed. Reversible (re-symlink / `bun run setup`).

**Lessons**:
- **A loader change can silently break a previously-working command without touching the file.** When `/x` 404s but the file is valid: suspect *discovery* (stale cache → `/reload-skills`; symlinks; a duplicate name across global+project), not the file.
- **Symlink-as-single-source-of-truth is fragile across tool versions.** For anything a tool *discovers* (commands/skills), prefer real files or have `setup` write copies — the DRY symlink saves nothing if the discovery pass skips it. Durable follow-up: make `bun run setup` write real command files so cloners get reliable slash-commands on the new loader.
- **The slash command is sugar, not the engine.** Builder Mode is entered by typed intent ("build X, go") via CLAUDE.md routing — a flaky command loader never actually blocks a user. Worth stating out loud when reassuring.

## Hardcode a "never": instructions + memory + a technical guard hook (don't rely on compliance alone)

**Setting**: Hamza asked to "hardcode" two forever-rules across every session/agent/dispatch — **Hamzaish is forever public** and **our conversations never go in a repo** — plus a nuanced visibility policy for *other* repos.

**What we built (defense in depth)**:
- **Instruction layer** — global `~/.claude/CLAUDE.md` "Two Absolute Invariants" (loaded in every session, project, and dispatched/background agent).
- **Memory layer** — dedicated recalled memories (`hamzaish-always-public`, `conversations-never-in-repo`), pinned in `MEMORY.md`.
- **Enforcement layer** — a `PreToolUse(Bash)` guard hook (`~/.claude/hooks/guard-repo-visibility.sh`) that *physically blocks* visibility changes: Hamzaish→private/archive = hard block (no agent override); any other PUBLIC→private/archive = blocked unless Hamza double-confirmed (marker `HAMZA_CONFIRM_PRIVATE=yes-twice` / token `I-DOUBLE-CONFIRM-PRIVATE`). Tested 11 cases (6 allow incl. `gh repo create --private` & make-public, 5 block) *before* registering.

**Lessons**:
- **A rule that must NEVER be violated deserves a technical block, not just prose.** Instructions rely on the model obeying; a hook is deterministic. Pattern: *invariant → `PreToolUse` hook that exits 2 on the dangerous pattern* (mirrors the existing AskUserQuestion-recommendation guard).
- **Gate, don't forbid, the reversible-but-costly case.** Other-repo public→private isn't "never" — it's "rare + double-confirmed," so the hook blocks-with-an-override-marker rather than hard-blocking. The marker is the technical proof a deliberate double-confirm happened.
- **Test a global Bash hook exhaustively before wiring it.** It intercepts EVERY command — a false-positive would wedge every session. Allow-cases first, fail-open by default, block only on a precise matched pattern.

**Decision**: visibility policy is now — repos toggle freely until first public; public→private needs Hamza's gated double-confirm; **Hamzaish is the absolute never**, enforced by the guard hook.
