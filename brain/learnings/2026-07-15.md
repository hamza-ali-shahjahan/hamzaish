# Learnings — 2026-07-15

## A fix verified only by mocks is not verified — the Sentry fix was wrong for two hours and every test was green

The starter shipped `@sentry/nextjs`, `withSentryConfig`, and `NEXT_PUBLIC_SENTRY_DSN` with no `instrumentation.ts`, so `Sentry.init()` was never called: an observability costume capturing zero errors. The fix added `instrumentation.ts` at the **project root** — and the mocked unit test (import `register()`, assert `init` called) passed, `typecheck` passed, `next build` compiled successfully. All green, still broken: this app lives in `src/app/`, so Next.js only loads `src/instrumentation.ts` and **silently ignores a root-level one** — no warning, no error, no output file. The bug was caught only by booting the built app against a fake Sentry ingest server and firing a real error: `flush()` returned `false` and zero envelopes arrived. After moving the file to `src/`, the same probe delivered an envelope carrying the canary nonce.

The mocked test could never have caught it, because it imports the module directly and thus *proves the module works, not that anything loads it*. The generalizable rule: **a test that imports the thing under test cannot prove the framework wires it up — for any convention-over-configuration hook (file-location contracts like `instrumentation.ts`, `middleware.ts`, `route.ts`, `sitemap.ts`), the assertion must be made against the running system, or on the file's location.** Both were added: `verify-live.ts` A11 (live) and `sentry-instrumentation-location.test.ts` (cheap, every commit). This is the same shape as 2026-07-14's *"pattern-match is a hypothesis, not evidence"* — the fix looked right, matched the docs from memory, and was wrong.

## `check-starter` says "launch-ready" about a scaffold whose `typecheck` and `build` both fail

`bun run check-starter` prints *"✓ starter is launch-ready: a brand-new scaffold works from zero"* — and it only runs `bun install`. A pristine scaffold today fails `bun run typecheck` (10 errors: `@supabase/ssr` 0.5.x overload resolution picks the deprecated `get/set/remove` signature, so `setAll`'s param is an implicit `any` under `strict`) and fails `bun run build` in at least three places (`typedRoutes: true` + `<Link href="/privacy">`/`/terms` routes that don't exist + `href={t.href}` typed as `string` in `pricing/page.tsx`). The starter's own CI gates `typecheck → lint → test → build`, so **every scaffolded product's first CI run is red at steps 3 and 5 of 5** — the gate is real and has never been green, because it only ever runs inside a scaffolded product's repo, never here.

The guard's *claim* ("launch-ready") is far broader than its *assertion* ("bun install exited 0"), and the gap went unnoticed because the claim was the thing anyone read. Rule: **a guard's success message must state what it actually asserted, not what the assertion is a proxy for.** `check-starter --boot` exists and would have caught the build break; nothing runs it. The cheapest real fix is to run the starter's own `typecheck`/`build` in the factory's CI against a scaffold — the template is the artifact that ships every time, so it deserves the ratchet more than the markdown does.

## The factory's four unit-test files never ran, and a bare `bun test` still can't run them

Wired `bun test` into CI (`.github/workflows/ci.yml`) — 36 tests across 4 files that no pipeline had ever executed. It required scoping to `./factory ./scripts ./dashboard`: a bare `bun test` walks into `references/` (study-only clones of other people's repos that fail without their own build steps) and `templates/`, whose vitest suites bun's runner cannot execute at all (`vi.resetModules`/`vi.unstubAllEnvs` are missing from bun's compat shim). Positional filters match on **path substring**, so `bun test scripts` also matches `references/openclaw/src/scripts/` — anchoring with `./` is load-bearing, not cosmetic. Rule: **when a repo vendors other people's repos, every repo-wide tool invocation needs an explicit scope, and the scope belongs in the committed script, not in someone's memory.**
