# 0001 — Pull unverified coverage stats from homepage + OG card

- **Date**: 2026-06-07
- **Decision**: Removed all headline dataset figures from outward copy and reframed "Coverage" to describe the *sources* rather than totals. Specifically: homepage `sources` no longer quotes `9M+` case-law opinions, `1976→` / "full patent record" patents, or `22M+` copyright registrations; the copyright row was deleted entirely; the H2 changed from "Live US case law and the full patent record" to "Built on public US case law and patent data"; copy now says coverage is "growing through beta." The OG/Twitter card footer changed from "9M+ opinions · full patent record · 22M+ registrations" to "Research · Clearance · Daily digests."
- **Why**: A live-DB check (Neon, 2026-06-07) showed the claims were materially false: `case_law_index` = 775 rows (not 9M+), `patent_index` = 33,676 rows covering grant dates 2025-02→2026-04 only (not "1976→" or "full"), and **no copyright/registration table exists at all** (the "22M+ since 1978" claim had zero data behind it). `citation_edges` = 0. These violate Hamzaish operating principle #13 (honest copy — never claim what isn't true). For a decision-support tool, one disprovable claim poisons every true one.
- **What would prove it wrong**: If the index actually reaches a size worth quoting AND we keep the number continuously true (a live count, not a hardcoded string), showing real totals would be fine and good. The principle bans *unverifiable/false* numbers, not numbers.
- **Revisit trigger**: When the DB is enriched to a meaningful scale (see the parallel enrichment work) — at that point, replace prose with a live, query-backed count component so the figure can never drift from reality. Also revisit if/when copyright-registration data is actually ingested (then the copyright row can return, honestly).
